## 개요

rgb Managers는 프로젝트 관리, 고객·견적서 관리, 고객 문의 관리, 자사 홈페이지 콘텐츠 관리 등으로 구성된 내부용 백오피스 서비스입니다. 기존의 PHP 기반 프로젝트 관리용 게시판을 Next.js, MongoDB 기반으로 리뉴얼하는 것으로 시작하여, 사내에서 필요한 도구들을 확장해가며 운영해왔습니다.

이 프로젝트의 가치는 사내 구성원들이 하나의 데이터를 바라보고 업무를 수행하는 SSOT에 있습니다. 운영진/영업팀 PC에 흩어져 관리되던 고객 데이터와 견적서 정보, 매번 구두로 요청하거나 파일 위치를 물어가며 접근하던 데이터를 웹상에서 한눈에 파악할 수 있게 되었으며, 프로젝트 맥락과 히스토리 또한 구성원들이 쉽게 공유할 수 있게 되었습니다. 또한 분산·중복 관리되던 자사 홈페이지 콘텐츠를 흡수하여 관리 포인트를 1/4 수준으로 절감하였습니다.

이 프로젝트를 통해 Next.js 기반 서비스 개발, 백엔드, DB, 배포·운영을 사내 실사용 환경에서 검증하였고, 이 과정에서 정리한 관리 화면 구조와 운영 경험은 이후 운영자 CMS와 관리 시스템 구축의 기준점이 되었습니다.

## 작업범위

- UI/디자인, FE, DB 설계, 마이그레이션, 배포 및 운영
- Next.js 13 기반 신규 구축, Next.js 15·shadcn 기반 리뉴얼
- MySQL 레거시 데이터 MongoDB 마이그레이션
- Ubuntu, Nginx 기반 운영 환경 구축

## 주요 기능

- 역할 기반 접근 제어
- 프로젝트 정보, 진행 단계, 담당자, 댓글 관리
- 고객 정보 관리, 엑셀 대량 업로드·다운로드
- 견적서 작성, 실시간 미리보기, PDF 출력, 견적서 템플릿 관리
- 자사 홈페이지 포트폴리오·문의 통합 관리

## 핵심 설계 결정

### 사내 환경과 개발 자원을 고려한 데이터베이스 선택

데이터베이스는 사내 환경과 개발 자원을 기준으로 결정했습니다. 당시 요구사항은 계속 바뀌었고, 전담 백엔드·DB 개발자가 없는 상황에서 이후 유지보수도 JS 기반 개발자가 맡을 가능성이 높았습니다.

- 조직 문화와 업무 환경상 필드 추가·변경이 잦음.
- 전통적 백엔드/DB 개발자의 부재로 복잡한 관계형 모델 운영 부담이 큼.
- 초기에는 고객, 견적, 프로젝트, 홈페이지 콘텐츠를 빠르게 흡수하는 것이 더 중요함.

따라서 러닝커브를 낮추고 스키마 변경에 유연하게 대응하기 위해 MongoDB를 선택했습니다. 이후 기능이 늘어나면서 담당자, 고객, 견적, 프로젝트 사이의 관계가 강해졌고, 이 지점에서는 관계형 모델이 더 적합한 영역도 있다는 점을 확인했습니다.

### 직접 구현한 UI를 shadcn/ui 기반 구조로 전환

최초 버전에서는 MUI 같은 완성형 UI 라이브러리도 검토했지만, 사내 디자인 톤에 맞춘 커스텀 필요성이 크고 화면 구조를 직접 통제해야 했기 때문에 폼, 테이블, 사이드시트 등 주요 UI를 직접 구현했습니다.

운영 기간이 길어지면서 직접 구현한 UI를 계속 유지보수하는 비용과, 새로운 UI가 필요할 때마다 새로 개발해야 하는 부담이 증가했습니다. V2에서는 Tailwind CSS와 shadcn/ui 기반으로 공통 컴포넌트 구조를 다시 정리해 디자인 요구사항을 충족하면서도 UI 구현 비용을 줄였습니다. 그 결과 개발자는 반복 UI보다 도메인 기능 구현에 더 집중할 수 있었습니다.

### HTML/CSS 기반 프리뷰를 Puppeteer로 PDF 렌더링

견적서 관리 모듈에는 작성된 견적서를 PDF로 내보내는 기능이 포함되어 있습니다. 이 기능은 아래 요구사항을 함께 만족해야 했습니다.

- 회사의 까다로운 디자인 기준을 반영한 견적서 양식 구현.
- 작성 화면에서 입력 내용이 즉시 반영되는 실시간 프리뷰 제공.
- 회사의 공식 반출 문서로 사용할 수 있는 고해상도 PDF 생성.

디자인 요구사항을 최대한 정확히 반영하기 위해 견적서 문서는 HTML/CSS 기반 컴포넌트로 구현했고, 폼 상태를 프리뷰에 연결해 작성자가 출력 결과를 보며 수정할 수 있게 했습니다.

PDF 변환은 해상도 문제로 파노라마 필름 OMS와 brand에서 사용했던 html2pdf.js 기반 클라이언트 변환 방식이 적합하지 않았습니다. 대신 HTML/CSS 기반의 복잡한 디자인을 그대로 렌더링하면서 고품질 PDF를 생성할 수 있도록 Puppeteer를 사용해 서버에서 처리했습니다. 서버 내 폰트와 브라우저 의존성, PDF 생성 시 부하가 있었지만, 사용 빈도와 프로젝트 규모상 수용 가능한 비용이라고 판단했습니다.
