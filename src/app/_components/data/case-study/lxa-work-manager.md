## 개요

Work Manager는 제 프리랜서 업무의 고객, 견적, 프로젝트, 일정, 수금, 파트너 정산을 정규화해 관리하기 위해 제작한 개인용 업무 시스템입니다. 1인 외주 업무에서 고객·견적·정산 데이터가 흩어지며 생기는 관리 비용을 줄이는 것이 목표였습니다.

직접 개발해둔 어드민 프레임워크를 기반으로 AI 에이전트를 활용해 구현했고, 현재 실제 업무에 사용하며 기능을 확장하고 있습니다.

## 작업범위

- 고객, 견적, 프로젝트, 일정, 정산 업무 흐름 정의
- 어드민 프레임워크 기반 관리 UI 구조 확장
- 워크플로 기반 데이터 모델 설계
- Docker PostgreSQL, LaunchAgent, 백업, 리빌드 스크립트 기반 로컬 운영환경 구성

## 주요기능

- 고객 정보 관리
- 버전 기반 견적서 관리, PDF 견적서 출력
- 프로젝트별 청구, 수금, 지출, 외주지급 장부
- 프로젝트 일정, 간트, 투두, 메모 위젯
- 프로젝트·고객별 파일 관리 드라이브

## 설계 포인트

### 로컬 상시 구동에 맞춘 스택 재구성

어드민 프레임워크를 활용해 Supabase와 Next.js 기반 프로토타입을 빠르게 제작하고, 실사용을 거쳐 React 클라이언트, Hono API 서버, Drizzle ORM, PostgreSQL 구조로 재구성했습니다. 개인 PC에서 상시 구동하는 업무 도구에 Supabase 풀패키지를 계속 실행하는 것은 과하다고 판단했고, 실제로 필요한 것은 PostgreSQL 데이터 저장소와 조인·트랜잭션이 얽힌 업무 흐름을 다룰 API 서버와 ORM이었습니다.

### 견적·프로젝트 중심 데이터 모델 설계

견적서는 고객 피드백에 따라 금액과 범위가 여러 번 바뀌기 때문에, 하나의 견적 번호 아래 여러 버전을 보존하고 계약 시점에 최종 버전을 지정하는 구조로 설계했습니다. 계약 상태에서만 최종 버전이 존재하도록 DB 제약으로 보장했습니다. 프로젝트 전환 후에는 청구, 수금, 지출, 외주지급을 분리해 계약 이력과 정산 흐름을 추적하도록 했습니다.

핵심 과제는 견적서와 프로젝트를 어느 정도까지 연결할지였습니다. 견적에서 프로젝트가 생성되는 일반적인 흐름만 보면 최종 견적 버전을 강하게 참조하는 구조가 자연스럽지만, 실제 업무에서는 견적 없이 시작되는 프로젝트, 계약 이후의 범위 변경, 청구 대상 분리 같은 예외가 생깁니다. 그래서 프로젝트는 견적 이력을 추적할 수 있게 연결하되, 진행 상태와 정산 정보는 프로젝트 자체 데이터로 독립해 관리하도록 분리했습니다. 컴포넌트를 과하게 결합하면 예외 처리가 어려워지는 프론트엔드 설계와 같은 문제로 보고, 정합성과 유연성 사이에서 결합 수준을 조절했습니다.

### 간트 타임라인 인터랙션 사양 설계

프로젝트 일정 화면의 간트 타임라인은 기성 라이브러리를 검토한 뒤 직접 구현으로 방향을 정했습니다. 매일 쓸 도구라 원하는 동작이 명확했고, 레인 구조와 상호작용을 그 수준까지 맞추기에는 라이브러리가 정한 프레임이 제약이 된다고 판단했습니다.

인터랙션 사양은 전부 직접 정의했습니다. 바를 드래그하는 동안은 픽셀 단위로 따라오다 놓는 순간 날짜에 스냅되는 이동, 클릭과 드래그의 구분, 이동 방향에 따라 수평·수직을 분리하는 축 잠금, Alt 드래그 복제의 고스트 바 프리뷰 같은 동작들입니다. 구현은 AI 에이전트를 활용하고, 정의한 사양대로 동작하는지 실사용으로 검증하며 어긋나는 동작을 반복 수정해 완성했습니다.

### 개인 PC 대상 로컬 운영환경 구성

외부 사용자가 없는 개인 업무 도구라 클라우드 배포 대신, 개인 PC에서 안정적으로 상시 실행되고 데이터와 파일이 로컬에 남는 구성을 택했습니다. Docker PostgreSQL로 데이터 저장소를 고정하고, Hono 서버가 API와 빌드된 React 정적 파일을 함께 서빙합니다. macOS LaunchAgent로 부팅 시 자동 실행되게 하고, 종료·재빌드·백업 스크립트로 운영 작업을 명령어화했습니다.
