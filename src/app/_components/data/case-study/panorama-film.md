## 개요

솔라가드 파노라마 필름 브랜드 홈페이지, 시공점 발주 관리 서비스(OMS), 운영 어드민을 구축한 프로젝트로, 서비스 오픈 후 약 2년 간 전국 72개 시공점을 대상으로 운영 및 유지보수되고 있습니다.

본사·총판·시공점 계층 간의 상품 발주, 보증서·시공내역서 발급, 2가지 핵심 요구사항을 중심으로 프로젝트를 설계 및 구현하였습니다.

## 작업 범위

- 브랜드 홈페이지 반응형 UI 구현
- 시공점 찾기, 게시판, 보증서 조회 기능 구현
- 시공점 발주, 장바구니, 주문서 작성, 보증서 발급 화면 구현
- 지역거점·시공점 등급에 따른 상품 노출, 주문 가능 액션, 관리 화면 흐름 분기
- 회원, 상품, 주문, 보증서, 게시판, 콘텐츠 관리 어드민 구현
- Supabase 기반 데이터 구조, 서버 액션, 배포 환경 구성

## 주요 기능

- 시공점 상품 발주, 장바구니, 주문 관리
- 보증서 발급, 조회, PDF 다운로드·인쇄
- 발주·보증서 카카오 알림톡 발송
- 매출 통계, 회원·상품·게시판·콘텐츠 관리 어드민
- 브랜드 사이트 내 시공점 찾기, 게시판, 보증서 조회

## 핵심 설계 결정

### 최종 상품 주문 화면에서의 상태 유지

상품 주문 화면은 사용자 의사결정의 마지막 단계입니다. 마지막 문턱에서 의도치 않게 '처음부터 다시'라는 불쾌한 경험을 방지하는 것이 중요하다고 판단, 주문 상태를 관리하는 흐름을 설계하였습니다.

- 상품 상세, 장바구니 두 가지 경로에서 주문 화면으로 진입할 수 있도록 주문 데이터 모델 설계
- 두 경로에서 전달된 주문 상태를 주문 화면에서 유지하는 방식 분석
  1. **URL Parameter**: 새로고침에는 대응하기 쉽지만, 주문 데이터가 길어지고 URL 직접 수정으로 의도와 다른 주문 상태가 만들어질 수 있음
  2. **React State**: 페이지 이동과 새로고침에 취약하고 상태를 유지하기 위해 불필요한 상위 구조가 필요함
  3. **sessionStorage/localStorage**: 새로고침 복구는 가능, 주문 화면에서 다시 React 상태로 변환하고 수정 사항을 저장소와 동기화해야 하는 중복 관리 발생
  4. **Zustand persist**: 데이터를 하나의 클라이언트 상태로 관리하면서, 삭제·수량 변경과 새로고침 복구를 같은 레이어에서 처리 가능
- 최종적으로 Zustand persist를 사용해 주문 화면의 상품, 수량 상태를 관리했고, 주문 완료 후 해당 상태를 초기화

### 화면 UI와 문서 출력물의 요구사항을 분리

보증서·시공내역서는 조회, PDF 다운로드 및 인쇄가 가능해야 했습니다. 본 프로젝트에서는 HTML/CSS로 문서를 구현한 뒤, 클라이언트에서 html2pdf.js로 PDF를 생성했습니다. 이 방식의 한계점을 운영 중 확인했고, 후속 프로젝트의 유사 사례에서 개선된 방향을 적용하였습니다.

- HTML/CSS 기반 구현으로 보증서 디자인 반영 용이성 확보
- html2pdf.js 기반 클라이언트 변환으로 다운로드·인쇄 기능 구현
- 운영 이후 해상도, 페이지 규격, 인쇄 출력 품질의 한계 확인
- 후속 프로젝트 유사 사례에서 Puppeteer 기반 서버 렌더링과 @react-pdf/renderer 방식 검토·적용

### 어드민 화면의 반복 구조를 재사용 가능한 패턴으로 정리

회원, 상품, 주문, 보증서, 게시판 관리는 서로 다른 도메인이지만 목록, 검색, 상세, 수정, 상태 변경이라는 화면 패턴을 공유했습니다. 이 반복 구조를 기준으로 어드민 UI를 구성해 이후 유사한 관리 시스템에서 재사용할 수 있는 구조를 설계했습니다.

- 목록 테이블, 검색·필터, 상세, 등록·수정 폼의 반복 구조 분리
- UI 레이아웃과 컴포넌트, 데이터 및 상태 관리 레이어를 패턴으로 설계
- 상품·주문·보증서 관리 화면에서 공통 컴포넌트와 화면 구성 규칙 재사용
- 해당 어드민 패턴을 시작으로 후속 프로젝트들의 어드민 앱의 구조로 지속·개선

## 트러블슈팅과 운영 한계

### PostgREST 조회 한계 대응

운영 중 테이블 데이터를 엑셀로 내려받는 기능에서 1000건을 초과하는 데이터가 누락되는 버그가 리포트되었습니다. PostgREST 조회시 1000건 제한이 원인임을 확인하였고, 테이블 행 count와 1000건 기준으로 쿼리를 반복하여 전체 데이터를 뽑도록 하였습니다.

### 배포 환경 제약 대응

카카오 알림톡 API가 Vercel의 서버리스 환경에서는 고정 IP 문제로 사용할 수 없음을 인지하였습니다. 고객사에게 알림톡 API용 마이크로 서버를 구축할 것을 제안하였고, 관리 포인트의 증가가 우려되어 최종적으로는 가비아 클라우드 서버에 self-hosting으로 프로젝트가 배포되었습니다.

## 결과와 배운 점

이 프로젝트는 회사가 기존 기업형 홈페이지 제작에서 Next.js 기반 웹앱 제작으로 범위를 확장하는 첫 사례가 되었습니다. 개발과 배포·운영 과정에서 발생한 이슈를 문서화해 이후 프로젝트의 대응 기준으로 남겼고, 보증서 발급과 어드민 화면처럼 반복 가능성이 높은 구조를 정리했습니다.

이 케이스를 기반으로 같은 스택의 프로젝트 6건을 추가로 진행했으며, 어드민 화면 구조와 운영 대응 경험은 후속 고객사 프로젝트와 자사용 서비스에도 적용하며 개선했습니다.
