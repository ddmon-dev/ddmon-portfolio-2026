## Overview

솔라가드 한국본사가 프리미엄 자동차 필름 브랜드 '파노라마 필름'을 런칭하며 구축한 B2B 웹 서비스입니다. 브랜드 사이트, 시공점 전용 발주·보증서 발급 시스템(OMS), 운영 어드민 — 3개의 Next.js 앱으로 구성되며, 런칭 후 전국 100곳 이상의 시공점이 사용하고 현재까지 직접 유지보수하고 있습니다.

회사의 첫 Next.js 고객사 프로젝트로, 재사용 가능한 공통 구조와 모듈을 후속 프로젝트의 표준 기반으로 설계했고, 이 기반 위에서 7건의 Next.js 프로젝트가 추가 수주·납품되었습니다.

## 작업범위

- 1인 단독 개발 — 3개 앱, 약 56개 화면, 4개월
- 기획 문서·시안 없이 OMS·어드민 기능명세를 직접 작성해 요구사항·정책 정의
- OMS·어드민 화면 설계와 FE 개발, 브랜드 사이트 반응형 구현(디자인 시안 기반)
- 백엔드 개발자 없이 DB 스키마·Supabase 서버 로직까지 직접 책임
- 배포 환경 구성과 운영·유지보수 지속
- 프로젝트 관리·고객사 커뮤니케이션

## Key Features

- 상품 발주 · 장바구니 · 발주 관리, 주문번호 자동 생성
- 시공 필름 보증서 발급·조회, 보증서 PDF 다운로드·인쇄
- 발주·보증서 카카오 알림톡 발송
- 매출 통계, 회원·상품·게시판·콘텐츠 관리 어드민

## Challenges

### 비용 기준으로 그은 추상화 경계 — 관리 화면 재사용 구조 설계

어드민의 반복되는 목록·검색·페이지네이션·작성/수정 골격을 config 기반 제네릭 구조로 추상화하면서, 추상화를 어디서 멈출지를 가장 중요한 설계 결정으로 삼았습니다. 하나의 제네릭이 모든 케이스를 흡수하게 하면 컨테이너 API와 러닝커브, 커스텀 비용이 함께 치솟는다고 봤습니다 — 잘못된 추상화는 중복보다 비싸기 때문입니다. 그래서 일반 화면은 공용 구조로 찍고(직접 사용 7개), 추가 조인·도메인 분기가 필요한 화면은 같은 재사용 인프라 위에서 복제해 커스텀했습니다.

서버 컴포넌트가 데이터를 조회해 컨텍스트로 주입하고, 모달 폼의 생성/수정·파일 업로드·라이프사이클은 전용 훅으로 캡슐화했으며, 필터·페이지네이션은 URL 쿼리스트링을 단일 소스로 쓰도록 표준화했습니다. 첫 Next.js 고객사 프로젝트를 다음 프로젝트가 올라설 기반으로 의식하고 설계한 결과, 이 구조는 후속 7개 프로젝트의 출발점이 됐고 이후 TypeScript·선언형 폼으로 정식 재설계한 어드민 템플릿으로 이어졌습니다.

### 클라이언트를 신뢰하지 않는 설계 — 발주 정합성을 서버까지 보장

상품 → 장바구니 → 주문 → 완료 멀티스텝 발주에서 클라이언트 표시 상태와 서버 권위 데이터를 분리해 설계했습니다. 주문 작성 상태는 클라이언트 상태값(새로고침 시 유실)·URL 파라미터(주소로 주문 조작 가능)·영속 스토어 세 안을 비교해, URL보다 노출이 적은 localStorage 영속 스토어로 입력 유실을 막았습니다.

가격과 포인트는 클라이언트를 신뢰하지 않았습니다. 백엔드 개발자가 없는 환경에서 화면이 약속한 결과를 끝까지 책임지려면 검증이 서버에 있어야 한다고 보고, 주문 제출 시 서버가 회원 등급별 가격으로 재계산·검증해 불일치하면 주문을 중단하도록 했습니다. 진짜 트랜잭션이 없는 멀티테이블 쓰기는 실패 시 보상 처리로 되돌리고, 비가역 작업인 알림톡 발송은 주문 확정 이후로 분리해 핵심 흐름과의 의존을 끊었습니다. 캐싱은 가격이 시공점 화면에 지연 반영되는 리스크가 이득을 넘는다고 보고 온디맨드 갱신으로 일관시켰습니다.

### 미팅으로 익힌 시공 도메인을 입력 단계 불변식으로 모델링

자동차 필름 시공 보증서를, 한 차량의 유리 위치별로 서로 다른 필름과 가격을 한 장에 묶는 매트릭스형 폼으로 설계했습니다. 기획서도 시안도 없이 고객사 미팅만으로 해석한 도메인을, 화면에서 깨질 수 없는 입력 규칙으로 옮긴 것이 핵심입니다.

한 유리 위치는 한 번만 배정되도록 이미 선택된 위치를 선택지에서 제외하고, 실제 시공가는 정가 총액을 넘지 못하게 묶었으며, 부위별 단가에서 정가·할인·실시공가를 실시간으로 파생했습니다. 발급 횟수가 부족하면 발급 자체를 막아, 데이터가 저장되기 전에 도메인 규칙이 입력 단계에서 강제되도록 했습니다.

### 고정 IP 제약을 최소 범위로 좁힌 배포 아키텍처 제안

카카오 알림톡 API의 발신 서버 고정 IP 요건으로 Vercel 배포를 유지할 수 없게 되자, 알림톡 발송 서버만 분리하고 앱은 Vercel에 유지하는 최소 범위 전환을 먼저 제안했습니다. 고객사가 인프라 일원화를 택하면서 가비아 클라우드 셀프호스팅으로 전환했고, Nginx Proxy Manager 리버스 프록시와 배포 환경을 직접 구성해 오픈했습니다.
