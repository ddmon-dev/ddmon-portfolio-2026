## Overview

프리미엄 자동차 필름 브랜드 '파노라마 필름' 런칭에 맞춰 구축한 B2B 웹 서비스입니다. 브랜드 사이트, 전국 시공점이 쓰는 발주·보증서 시스템(OMS), 운영 어드민까지 3개의 Next.js 앱으로 구성됩니다. 백엔드 개발자가 없는 환경에서 발주·보증서 업무 흐름을 화면 설계부터 데이터 구조까지 풀어냈습니다.

## Highlights

- 브랜드 사이트 15개 · 발주 시스템 23개 · 어드민 17개 — **3개 Next.js 앱, 총 55개 화면을 4개월에 단독 개발**해 오픈했습니다.
- 발주 시스템과 어드민 40개 화면은 기획서·시안 없이 **동작하는 화면을 만들어 제안하며 직접 설계**했습니다.
- 새로고침에도 유지되는 주문 폼, 즉시 반응하는 장바구니 — 발주 플로우의 사용자 경험을 **서버 검증과 실패 복구까지 챙겨 완성**했습니다.
- 오픈 후 **전국 100곳 이상의 지역거점·시공점이 사용** 중이고, 퇴사 후에도 고객사 직접 의뢰로 유지보수를 잇고 있습니다.

## Challenges

### 기획서도 시안도 없는 발주 시스템, 화면 설계부터 시작

브랜드 사이트는 디자이너 시안이 있었지만, 발주 시스템과 어드민은 요구사항 문서와 레퍼런스 사이트만 받은 채 "시공점이 본사에 필름을 발주하고 보증서를 발급하는 업무"를 화면으로 옮겨야 했습니다.

문서로 기획을 확정하는 대신, 고객사 미팅으로 업무 흐름을 파악하고 유사 커머스 UX를 참고해 **동작하는 화면을 먼저 만들어 제안하고 피드백으로 다듬었습니다**. 지역거점·시공점 2단계 권한 구조와 회원 유형별 가격 노출도 이 과정에서 화면에 담겼습니다.

### 발주 플로우 — 상태, 검증, 실패 복구까지 화면의 책임으로

상품 → 장바구니 → 주문 → 완료 사이에서 상태가 끊기지 않아야 했습니다. 장바구니는 수량 변경을 UI에 먼저 반영하고 DB 동기화는 300ms 디바운스로 묶었으며, 주문 화면은 Zustand persist로 새로고침에도 작성 내용을 유지합니다. 발주 직전 입력 유실을 막으려 요구사항보다 먼저 넣은 설계입니다.

화면이 "주문 완료"라고 말하면 데이터도 실제로 그래야 했습니다. 모든 DB 접근은 서버 전용으로 강제하고(브라우저용 Supabase 클라이언트는 에러를 던지게 차단), 주문 제출은 Server Action에서 DB 가격으로 재검증한 뒤 실패하면 주문을 되돌리는 보상 처리를 거칩니다 — Supabase SDK에 트랜잭션이 없어 RPC(Postgres 함수)가 정석이지만, 당시 Postgres 숙련도로는 유지보수 리스크가 크다고 판단했습니다. 되돌릴 수 없는 알림톡 발송은 주문이 완전히 성공한 뒤에만 실행됩니다.

캐싱은 의도적으로 쓰지 않았습니다. 어드민에서 바꾼 가격이 시공점 화면에 늦게 보이는 리스크가 이 규모에서 캐싱의 이득보다 크다고 보고, 변경 시 revalidatePath로 즉시 갱신했습니다.

### 17개 관리 화면을 하나의 공통 구조로

어드민은 목록·검색·페이지네이션·작성/수정 골격이 화면마다 반복됩니다. 서버 컴포넌트가 데이터를 조회해 ListProvider / WriteProvider 컨텍스트로 하위 UI를 감싸는 **제네릭 공통 구조로 17개 화면을 같은 패턴으로 처리**해, 납품 후의 기능 추가·유지보수에 대비했습니다.

검색 필터·페이지네이션·시공점 찾기가 URL을 상태의 단일 소스로 쓰게 만든 useQueryString 훅, 모달 시스템, 폼 컨트롤은 자체 라이브러리(DD_LIB)로 묶었습니다. 이 자산들은 의도대로 이후 Next.js 프로젝트들의 출발점이 됐습니다.

## Key Features

- 상품 발주 · 장바구니 · 발주 관리, 주문번호 자동 생성
- PPF / 틴팅 보증서 발급, PDF 다운로드·인쇄, 브랜드 사이트 보증서 조회
- 발주·보증서 카카오 알림톡 자동 발송 (비즈톡 연동)
- 매출 통계, 상품·게시판·콘텐츠 관리 어드민

## Result

2024년 9월 오픈 후 현재까지 특별한 장애 없이 운영 중이며, 전국 100곳 이상의 지역거점·시공점이 사용하고 있습니다. 퇴사 후에는 고객사가 직접 연락해 와 지금도 유지보수를 맡고 있습니다. 알림톡 API의 고정 IP 요구에는 앱을 Vercel에 두고 알림톡 서버만 분리하는 안을 제안했지만, 고객사가 관리 포인트 단일화를 원해 가비아 클라우드 self-hosting으로 직접 전환했습니다.

처음으로 Next.js를 고객사 프로젝트에 적용한 사례였고, 이후 Next.js 고객사 프로젝트 8건이 이어졌습니다.
