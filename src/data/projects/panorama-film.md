## Overview

솔라가드 한국본사가 프리미엄 자동차 필름 브랜드 '파노라마 필름'을 런칭하며 구축한 B2B 웹 서비스입니다. 브랜드 사이트, 시공점 전용 발주·보증서 발급 시스템(OMS), 운영 어드민 — 3개의 Next.js 앱으로 구성되며, 런칭 후 전국 100곳 이상의 시공점이 사용하고 현재까지 직접 유지보수하고 있습니다.

회사의 첫 Next.js 고객사 프로젝트로, 재사용 가능한 공통 구조를 후속 프로젝트의 표준 기반으로 설계했고, 이 기반 위에서 7건의 Next.js 프로젝트가 추가 수주·납품되었습니다.

## 작업범위

- 1인 단독 개발 — 3개 앱, 약 56개 화면, 4개월
- 기획 문서·시안 없이 OMS·어드민 요구사항·화면을 직접 정의
- OMS·어드민 화면 설계와 FE 개발, 브랜드 사이트 반응형 구현(디자인 시안 기반)
- 백엔드 개발자 없이 DB 스키마·Supabase 서버 로직까지 직접 책임
- 배포 환경 구성과 운영·유지보수 지속
- 프로젝트 관리·고객사 커뮤니케이션

## 기획 의도

솔라가드가 프리미엄 브랜드를 새로 런칭하며 B2B 발주 업무 전체를 웹으로 구조화하려는 프로젝트였고, 기획자도 화면 시안도 없는 상태로 착수했습니다. 그래서 요구사항을 기능명세서로 직접 작성해 — 브랜드·OMS·어드민 세 갈래로 나눠 — 고객사에 정책을 확인받은 뒤 그 명세를 기준으로 개발했습니다. 회원 등급, 지역거점-시공점 계층, 등급별 차등 가격 같은 정책은 고객사 요구였고, 제 몫은 그것을 데이터 모델과 화면으로 구조화하는 일이었습니다.

동시에 이 프로젝트를 일회성 납품이 아니라 후속 Next.js 프로젝트가 올라설 기반으로 삼겠다는 목표가 있었습니다. 그래서 개발 내내 "다음 프로젝트에서도 재사용할 구조와 컴포넌트"를 의식하며 설계했고, 실제로 그 의도가 이후 수주로 이어졌습니다.

## Key Features

- 상품 발주·장바구니·발주 관리, 주문번호 자동 생성
- 시공 필름 보증서 발급·조회, 보증서 PDF 다운로드·인쇄
- 발주·보증서 카카오 알림톡 발송
- 매출 통계, 회원·상품·게시판·콘텐츠 관리 어드민

## 핵심 설계 결정

### 추상화 경계를 비용으로 그었다

어드민에 반복되는 목록·검색·페이지네이션·작성/수정 골격을 config 기반 제네릭 구조로 추상화하면서, '어디까지 제네릭으로 흡수할지'를 가장 중요한 결정으로 다뤘습니다. 하나의 제네릭이 모든 케이스를 흡수하면 컨테이너 API와 러닝커브, 커스텀 비용이 함께 치솟는다 — 잘못된 추상화는 중복보다 비싸다는 판단이었습니다.

- 공용 구조 직접 사용 7개 / 추가 조인·분기가 필요한 화면은 복제 커스텀 6개 / 완전 커스텀 3개로 경계 분할.
- 복제의 중복 유지보수 비용을 제네릭 강제의 러닝커브·커스텀 비용보다 싼 쪽으로 택한 의식적 트레이드오프.
- 모달 폼 생성/수정·변경분만 올리는 파일 업로드·언마운트 정리를 전용 훅으로 캡슐화, 작성 위젯들이 공유.
- 후속 7개 프로젝트의 출발점이자, 이후 TypeScript·선언형 폼으로 재설계한 어드민 템플릿의 모태.

### 클라이언트를 신뢰하지 않는 서버 권위 검증

발주·결제 정보는 클라이언트 값을 신뢰하지 않고 서버에서 다시 계산·검증하도록 설계했습니다. 백엔드 개발자가 없는 환경에서 화면이 약속한 결과를 끝까지 책임지려면, 비즈니스 로직을 가장 확실하게 강제할 수 있는 곳은 서버라고 봤기 때문입니다.

- 주문 제출 시 클라이언트가 보낸 식별자로 DB 재조회, 등급별 동적 가격 컬럼으로 재계산해 값이 어긋나면 주문 중단.
- 적립 포인트·잔액도 클라이언트 값 폐기 후 서버 기준 재계산·검증.
- 폐쇄형 B2B라 위험도는 낮지만, 가격 변조 가능성을 미리 인지하고 차단하는 쪽을 선택.

### 진짜 트랜잭션 대신 보상 처리를 택했다

주문·발급처럼 여러 테이블에 걸친 쓰기를, 진짜 트랜잭션이나 RPC가 아니라 Server Action 레벨의 보상 처리로 정합성을 맞췄습니다.

- 결제 포인트의 단일 원천이 Supabase Auth 사용자 메타데이터라 일반 SQL 트랜잭션 경계 밖 — RPC로 묶는 쪽은 유지보수 리스크가 더 크다고 판단해 보상 처리 선택.
- 주문 실패 시 생성된 주문 롤백(연결 항목은 FK CASCADE로 동반 삭제), 보증서 발급은 저장·포인트 차감을 동시 시도 후 한쪽 실패 시 교차 보상.
- 진짜 원자성이 아니라는 한계(부분 실패 가능성)는 인지한 선택 — 폐쇄형 B2B의 트래픽·동시성 수준에서 수용 가능하다고 판단.

### 주문 작성 상태 영속화 — 세 안의 비교

멀티스텝 발주에서 작성 중 상태를 어디에 둘지 세 안을 비교해 결정했습니다.

- 클라이언트 상태값: 새로고침 시 유실 — UX 저하가 커서 탈락.
- URL 파라미터: 새로고침은 방어되지만 주소로 주문 화면을 조작할 수 있어 탈락.
- localStorage 영속 스토어: 기존 전역 상태를 그대로 쓰면서 URL보다 노출이 적어 채택.

표시 상태(클라이언트)와 권위 데이터(서버)를 분리한다는 같은 결의 결정으로, 클라이언트는 입력 편의를 맡고 검증은 서버가 책임집니다.

### 생소한 시공 도메인을 입력 단계 불변식으로 모델링

자동차 필름 시공 보증서를, 한 차량의 유리 위치별로 서로 다른 필름과 가격을 한 장에 묶는 매트릭스형 폼으로 설계했습니다. 기획서도 시안도 없이 고객사 미팅만으로 해석한 도메인을, 화면에서 깨질 수 없는 입력 규칙으로 옮긴 것이 핵심입니다.

- 한 유리 위치는 1회만 배정 — 이미 선택된 위치는 선택지에서 제외.
- 실제 시공가는 정가 총액을 넘지 못하도록 제한, 부위별 단가에서 정가·할인·실시공가를 실시간 파생.
- 발급 횟수 부족 시 발급 자체를 차단 — 데이터 저장 전, 입력 단계에서 도메인 규칙 강제.

## 트러블슈팅 · 운영 한계 대응

### PostgREST 1000행 한계를 청킹으로 우회

엑셀 전체 추출에서 한 번에 가져올 수 있는 행이 1000개로 잘리는 PostgREST 기본 한계에 부딪혔습니다. range로 구간을 잘라 반복 조회한 뒤 합치는 유틸로 우회했고, 매출·보증서·주문·회원 4개 화면의 전체 추출이 같은 유틸을 공유합니다.

### 관계 컬럼 OR 제약을 서브쿼리 union으로

주문 전체검색에서 상품명·주문자처럼 관계(조인) 컬럼을 OR로 한 번에 거는 쿼리를 PostgREST가 지원하지 않는 제약을 만났습니다. 각 조건을 서브쿼리로 병렬 조회해 주문번호 집합을 합친 뒤(union), 그 집합으로 다시 IN 필터를 거는 방식으로 풀었습니다 — 쿼리 엔진의 한계를 진단하고 그 위에서 우회 경로를 설계한 사례입니다.

### 명령형 폼 브릿지에서 선언형 템플릿으로

초기 보증서 폼은 비폼 UI를 React Hook Form에 명령형으로 이어 붙이는 방식이었습니다. 동작은 했지만 권장 패턴이 아니라고 판단해, 이후 본인 IP인 어드민 템플릿에서 Controller 기반 선언형 폼과 zod 스키마 검증으로 재설계했습니다. 같은 문제를 시간차를 두고 다시 푼, 납품작(JS·명령형) → 본인 자산(TypeScript·선언형)의 성장 흔적입니다.

## 배포와 운영

카카오 알림톡 API가 발신 서버 고정 IP 등록을 요구해 Vercel 배포를 유지할 수 없게 되자, 알림톡 발송 서버만 분리하고 앱은 Vercel에 유지하는 최소 범위 전환을 먼저 제안했습니다. 고객사가 인프라 일원화를 택하면서 가비아 클라우드 셀프호스팅으로 전환했고, Nginx Proxy Manager 리버스 프록시와 배포 환경을 직접 구성해 오픈했습니다.

런칭 후 전국 100곳 이상의 지역거점·시공점이 사용하고 있고, 이 프로젝트의 구조를 기반으로 후속 Next.js 프로젝트 7건이 이어졌습니다. 퇴사 후에도 고객사가 직접 유지보수를 맡겨, 런칭 2년이 지난 지금까지 스키마 변경이 이어지는 운영 중 서비스입니다.
