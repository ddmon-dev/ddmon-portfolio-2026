## Overview

파노라마 필름은 시공점이 상품을 발주하고, 시공 보증서를 발급하며, 본사가 전체 운영을 관리하는 B2B 웹 서비스입니다. 브랜드 사이트보다 핵심은 OMS와 어드민이었고, 저는 시공점 업무 흐름을 화면 상태, 입력 모델, 서버 검증 경계까지 이어지는 구조로 설계했습니다.

이 프로젝트는 회사의 첫 Next.js 기반 고객사 프로젝트였습니다. 3개 앱·약 56개 화면을 4개월 동안 단독으로 구현했고, 런칭 후 전국 100곳 이상의 지역거점·시공점이 사용했으며 현재까지 유지보수가 이어지고 있습니다.

## 작업범위

- 브랜드 사이트, 시공점 OMS, 운영 어드민 3개 앱 개발
- OMS·어드민 요구사항 정리, 화면 흐름 정의, 기능명세 작성
- 시공점 발주, 장바구니, 주문, 보증서 발급 UI 설계
- 반복 관리 화면의 목록·검색·작성/수정 구조 설계
- 화면이 요구하는 데이터 구조와 서버 검증 로직까지 직접 구현
- 배포 환경 구성, 카카오 알림톡 연동, 운영·유지보수 대응

## 기획 의도

솔라가드 한국본사가 프리미엄 자동차 필름 브랜드를 런칭하면서, 시공점 발주와 보증서 발급 업무를 함께 웹으로 옮기는 프로젝트였습니다. OMS와 어드민은 기획서나 화면 시안이 없는 상태였기 때문에, 회원 등급, 지역거점-시공점 계층, 등급별 가격, 보증서 발급 규칙을 기능명세로 정리하고 고객사 확인을 거쳐 화면과 데이터 구조로 옮겼습니다.

처음부터 일회성 납품으로만 보지 않았습니다. 반복될 어드민·CMS 프로젝트의 출발점으로 삼기 위해, 목록·검색·작성/수정 같은 관리 화면 골격과 모달 폼 흐름을 다음 프로젝트에서도 가져갈 수 있는 형태로 정리했습니다.

## Key Features

- 시공점 상품 발주, 장바구니, 주문 관리
- 회원 등급·지역거점·시공점 계층 기반 가격/권한 처리
- 시공 필름 보증서 발급, 조회, PDF 다운로드·인쇄
- 발주·보증서 카카오 알림톡 발송
- 매출 통계, 회원·상품·게시판·콘텐츠 관리 어드민

## 핵심 설계 결정

### 발주 화면의 편의 상태와 서버 권위 데이터를 분리

멀티스텝 발주 화면은 사용자가 작업을 잃지 않게 돕되, 최종 가격과 주문 가능 여부는 서버가 다시 판단하도록 경계를 나눴습니다.

- 작성 중 발주 상태는 localStorage 기반 전역 상태로 보존해 새로고침에도 흐름 유지.
- URL 파라미터는 복원에는 유리하지만 주문 화면 조작 여지가 있어 제외.
- 주문 제출 시 클라이언트가 보낸 가격·포인트 값은 폐기하고 DB 기준으로 재계산.
- 프론트엔드는 입력 편의와 진행 흐름을 맡고, 비즈니스 권위는 서버 검증에 둔 구조.

### 시공 보증서 도메인을 매트릭스 입력 모델로 설계

시공 보증서는 한 차량의 유리 위치마다 서로 다른 필름과 가격이 들어가는 도메인이라, 일반 게시판 폼으로는 자연스럽게 표현하기 어려웠습니다.

- 한 유리 위치는 한 번만 선택되도록 이미 선택된 위치를 다음 선택지에서 제외.
- 부위별 필름 단가에서 정가·할인·실시공가를 실시간 파생.
- 실시공가는 정가 총액을 넘지 못하도록 입력 단계에서 제한.
- 발급 가능 횟수가 부족하면 저장 단계가 아니라 입력 흐름에서 발급 자체를 차단.

### 반복 관리 화면은 공통화하되 커스텀 비용이 커지는 지점에서 멈춤

어드민의 목록, 검색, 페이지네이션, 작성/수정 모달을 config 기반 공통 구조로 묶되, 모든 화면을 하나의 제네릭으로 밀어 넣지는 않았습니다.

- 일반 관리 화면 7개는 공통 List/Write 구조로 처리.
- 조인·검색·도메인 분기가 강한 화면 6개는 같은 패턴을 복제해 커스텀.
- 완전히 다른 업무 흐름은 별도 구현으로 분리.
- 중복 제거보다 유지보수자가 이해 가능한 추상화 경계를 지키는 쪽을 선택.

### 다중 테이블 쓰기는 원자성 한계를 인지한 보상 처리로 수용

주문·보증서 발급처럼 여러 테이블과 포인트 차감이 함께 움직이는 작업은 Server Action 레벨의 보상 처리로 정합성을 맞췄습니다.

- 포인트의 단일 원천이 Supabase Auth 사용자 메타데이터라 일반 SQL 트랜잭션 경계 밖에 위치.
- RPC로 묶는 선택지는 유지보수 리스크가 크다고 판단해 애플리케이션 레벨 보상 처리 채택.
- 주문 실패 시 생성된 주문을 삭제하고, 연결 항목은 FK CASCADE로 동반 정리.
- 진짜 원자성이 아니라는 한계를 폐쇄형 B2B의 트래픽·동시성 수준에서 수용한 선택.

### 납품 프로젝트를 다음 프로젝트의 제작 기반으로 남김

이 프로젝트에서 만든 어드민 구조와 폼·모달 패턴은 이후 Next.js 고객사 프로젝트의 출발점이 됐습니다.

- 파노라마 이후 여러 Next.js 기반 프로젝트에 관리 화면 구조를 복제·개선하며 적용.
- 초기 JavaScript·명령형 폼 패턴의 한계는 이후 TypeScript·선언형 폼 기반 어드민 템플릿으로 재설계.
- 한 번 만든 납품물을 버리지 않고 다음 프로젝트의 제작 속도와 품질을 높이는 기반으로 전환.

## 트러블슈팅 · 운영 한계 대응

### PostgREST 1000행 한계를 청킹으로 우회

엑셀 전체 추출에서 PostgREST 기본 조회 한계로 1000행까지만 내려오는 문제를 만났습니다. range로 구간을 나눠 반복 조회한 뒤 합치는 유틸을 만들고, 매출·보증서·주문·회원 화면의 전체 추출이 같은 경로를 공유하게 했습니다.

### 관계 컬럼 OR 검색을 서브쿼리 집합으로 재구성

주문 전체검색에서 상품명·주문자처럼 관계 컬럼을 한 번에 OR 검색하기 어려운 제약이 있었습니다. 조건별 서브쿼리로 주문번호 집합을 만든 뒤 합치고, 그 집합을 다시 IN 필터로 조회하는 방식으로 운영자가 기대하는 검색 경험을 맞췄습니다.

### 고정 IP 제약으로 배포 구조를 전환

카카오 알림톡 API가 발신 서버 고정 IP 등록을 요구해 Vercel만으로는 배포할 수 없었습니다. 알림톡 발송 서버만 분리하는 절충안을 먼저 제안했지만, 고객사가 인프라 일원화를 선택하면서 가비아 클라우드 셀프호스팅으로 전환했습니다.

## 배포와 운영

런칭 후 전국 100곳 이상의 지역거점·시공점이 사용했고, 이 프로젝트에서 정리한 구조는 이후 Next.js 고객사 프로젝트들의 출발점이 됐습니다. 퇴사 후에도 유지보수 요청이 이어져, 런칭 2년이 지난 지금까지 스키마 변경과 운영 대응을 계속 맡고 있습니다.
