## Overview

프리미엄 자동차 필름 브랜드의 런칭과 함께 신규 구축한 B2B 웹 서비스입니다. 브랜드 사이트, 시공점 발주·보증서 발급 시스템(OMS), 관리자 어드민 세 개의 Next.js 앱으로 구성되며, 전국 지역거점·시공점의 발주와 보증서 발급이라는 실제 업무 흐름을 웹 시스템으로 구조화했습니다. 디자인 시안은 브랜드 사이트에만 제공되었고, OMS와 어드민은 화면 설계부터 개발·배포·고객사 커뮤니케이션까지 단독으로 수행했습니다.

## Highlights

- 디자인 시안 없이 고객사의 업무 요구에서 출발해 OMS 23개·어드민 18개 화면을 직접 설계하고, 지역거점/시공점 2단계 권한과 회원 유형별 가격 구조를 데이터 모델과 화면으로 풀어냈습니다.
- 장바구니에서 주문 완료까지 이어지는 발주 플로우를 Zustand 기반 상태 설계로 구현하고, 화면이 보여주는 가격·포인트가 거짓이 되지 않도록 서버 검증과 보상 롤백까지 직접 챙겼습니다.
- 카카오 알림톡 API의 고정 IP 요구로 배포 환경을 Vercel에서 self-hosting으로 직접 전환해, 기능 축소 없이 납품을 완성했습니다.
- 2024년 8월 납품 후 현재까지 전국 100곳 이상의 지역거점·시공점이 사용하는 서비스로 운영 중이며, 기능 추가와 유지보수 의뢰가 계속 이어지고 있습니다.

## Challenges

### 시안 없이 발주·보증서 업무를 41개 화면으로 설계

브랜드 사이트와 달리 OMS와 어드민은 디자인 시안도 화면 기획서도 없었습니다. "시공점이 필름을 발주하고, 시공 후 고객에게 보증서를 발급한다"는 업무 요구만 있는 상태에서, 누가 어떤 권한으로 무엇을 보고 처리하는지를 화면으로 정의하는 일부터 시작해야 했습니다.

고객사 미팅과 인터뷰로 실제 업무 흐름 — 지역거점과 시공점의 관계, 발주 단위와 결제 방식, 보증서가 최종 고객에게 전달되는 과정 — 을 파악하고, 유사 발주·커머스 서비스의 UX 패턴을 참고해 화면을 구성했습니다. 그 결과를 지역거점/시공점 2단계 권한 구조와 회원 유형별 가격 체계로 정리하고, OMS 23개·어드민 18개 화면으로 구현했습니다. 디자인 시안을 받아 옮기는 것이 아니라 업무를 이해해야 화면이 나오는 프로젝트였고, 이 경험이 이후 시안 없는 시스템 화면 설계의 기준이 되었습니다.

### 장바구니에서 주문 완료까지, 화면 간 상태 흐름

발주 플로우는 상품 선택 → 장바구니 → 주문서 작성 → 주문 완료로 화면이 이어지는 동안 수량·옵션·가격·적립 포인트가 어긋나지 않아야 했습니다. 상태를 어디에 둘지부터 판단이 필요했습니다. 장바구니는 Zustand 스토어로 관리하며 수량 변경을 UI에 즉시 반영하되 DB 동기화는 디바운스로 묶어 연속 조작에도 매끄럽게 동작하게 했고, 주문서 단계는 persist 미들웨어를 적용해 새로고침에도 작성 중인 주문 정보가 유지되도록 했습니다. 제출 검증은 React Hook Form으로 처리했습니다.

다만 화면 상태를 아무리 다듬어도 클라이언트가 보내는 가격은 신뢰할 수 없습니다. 화면이 보여주는 금액이 거짓이 되지 않도록 Server Action에서 DB 가격을 재조회해 검증하고, Supabase에 멀티테이블 트랜잭션 API가 없어 주문 저장 → 주문 상품 저장 → 포인트 차감을 순차 실행하며 실패 시 이전 단계를 되돌리는 보상 롤백을 구성했습니다. 알림톡 발송은 실패해도 발주 자체를 막지 않도록 분리했습니다. 프론트엔드 입장에서 낯선 데이터 구조와 비즈니스 로직이 사실 가장 어려운 부분이었지만, 화면과 사용자 경험을 끝까지 책임지기 위해 필요한 만큼 내려가 해결했습니다.

### 18개 관리 화면을 하나의 공통 구조로

어드민은 상품·주문·보증서·게시판·회원 등 18개 관리 화면이 필요했고, 모두 "목록 + 검색 + 페이지네이션 + 작성/수정"이라는 같은 골격의 반복이었습니다. 첫 Next.js 프로젝트였던 만큼 화면마다 코드를 복제하는 대신, 이후 프로젝트에서도 쓸 수 있는 구조를 만드는 데 가장 많은 고민을 들였습니다.

서버 컴포넌트가 데이터를 조회해 ListProvider/WriteProvider 컨텍스트로 하위 UI를 감싸는 제네릭 패턴을 만들어, 대상 테이블과 폼 필드만 바꿔 끼우면 새 관리 화면이 완성되도록 했습니다. 폼 컨트롤·모달 시스템·게시판 컴포넌트 같은 공통 요소는 자체 라이브러리(DD_LIB)로 분리했습니다. 의도대로 이 구조와 라이브러리는 백신정보포털 등 이후 Next.js 프로젝트들의 출발점이 되었습니다.

### 알림톡 고정 IP 요구, Vercel에서 self-hosting으로

개발 막바지에 카카오 알림톡(비즈톡) API가 발신 서버의 고정 IP 화이트리스트 등록을 요구한다는 사실을 확인했습니다. Vercel은 고정 IP를 제공하지 않았고, 발주·보증서 알림은 고객사의 핵심 요구라 뺄 수 없었습니다.

알림톡만 외부 서버로 우회하는 방법도 있었지만 구조가 복잡해진다고 판단해, 가비아 클라우드 서버에 Nginx Proxy Manager로 세 개 앱을 self-hosting하는 배포 환경을 직접 구성했습니다. 알림톡 인증 토큰은 12시간 캐싱해 불필요한 API 호출을 줄였습니다. 프로젝트 전체에서 가장 큰 돌발 이슈였지만, 배포 환경 전환으로 기능 축소 없이 납품을 마쳤습니다.

## Key Features

- 지역거점/시공점 2단계 권한 구조와 회원 유형별 가격
- 상품 발주, 장바구니, 발주 내역 관리
- 보증서(PPF/틴팅) 발급과 포인트 차감, 발급 내역 관리
- 브랜드 사이트 보증서 조회, PDF 다운로드와 전용용지 인쇄
- 발주·보증서 카카오 알림톡 발송
- 상품·게시판 관리, 매출 통계

## Result

2024년 8월 납품 이후 현재까지 특별한 장애 없이 운영 중이며, 전국 100곳 이상의 지역거점·시공점이 사용하고 있습니다. 납품 후에도 기능 추가와 유지보수 의뢰가 이어져 지금까지 직접 수행하고 있고, 이 프로젝트에서 만든 화면 구조와 공통 라이브러리는 이후 Next.js 프로젝트들의 기반이 되었습니다.
