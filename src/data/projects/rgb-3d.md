## Overview

알지비커뮤니케이션즈가 편집·그래픽 디자인에서 3D 영상 제작으로 사업을 넓히며 새로 만든 영상 브랜드 사이트입니다. 회사의 영상 작업물을 스크롤 인터랙션 중심으로 전시하고 온라인 문의를 받는 공개 사이트로, 기획부터 화면 설계·프론트엔드 구현·운영까지 단독으로 맡아 2024년부터 지금까지 운영하고 있습니다.

화려한 연출보다, 백엔드도 서버 관리자도 없이 혼자 만들고 운영하는 환경에서 데이터·트래픽을 먼저 따져 인터랙션을 설계한 것이 이 프로젝트의 무게입니다. 메인 히어로의 연출이 실제 포트폴리오 데이터로 이어지는 구조, 호버한 카드만 영상을 불러오는 로딩 방식처럼, 디자인이 요구한 인터랙션을 실제 데이터·운영과 어긋나지 않게 푸는 일이 핵심이었습니다.

## 작업범위

- 기획·프론트엔드·운영 1인 단독, 2024년 구축 후 현재까지 유지
- 디자인 시안은 메인 인트로·소개 페이지만 제공됐고, 포트폴리오 목록·카드·상세·문의 화면은 직접 설계
- 메인·소개 스크롤 인터랙션, 포트폴리오 브라우징 UX, 온라인 문의 파이프라인을 라이브러리 기본값이 아니라 직접 구현
- 백엔드·서버 관리자 없이 데이터 연동(MongoDB)·배포·서버 운영까지 직접

## 기획 의도

알지비가 3D 영상 제작으로 사업을 확장하며, 영상 작업물을 보여줄 새 브랜드 사이트가 필요했습니다. 기존 사이트를 잇는 게 아니라 영상 사업 전용 브랜드를 새로 세우는 일이었고, 회사 포트폴리오를 전시하고 문의를 받는 것이 목적이었습니다.

핵심은 정적인 전시가 아니라 스크롤과 호버에 반응하는 인터랙션으로 작업물을 보여주는 것이었습니다. 그래서 메인·소개 페이지의 스크롤 연출과 포트폴리오 카드의 확장 인터랙션을 사이트의 중심에 두고, 그 인터랙션이 실제 데이터·운영과 어긋나지 않게 구조를 잡는 데 무게를 실었습니다.

## Key Features

- 메인·소개 페이지 스크롤 인터랙션: 진행도에 따라 비주얼이 연속 변형
- 포트폴리오 목록: 카테고리 필터, 호버 시 확장되는 커스텀 팝오버(좌표 계산·화면 경계 보정, 썸네일→클립 영상 전환)
- 포트폴리오 상세: 영상 임베드 플레이어, 보던 목록 위치로 복귀
- 온라인 문의: 첨부파일 동봉, 접수 시 담당자 메일 발송

## 핵심 설계 결정

### 히어로 연출 카드를 실데이터로 채워 목록 시작점과 결합

메인 히어로의 확장 카드를 장식용 더미가 아니라 실제 포트폴리오 데이터로 채워, 스크롤하면 그대로 하단 목록의 시작점으로 이어지게 설계했습니다. 디자인이 "그래픽 요소처럼 보이는 카드가 실제 데이터와 결합되는" 형태였고, 이걸 어떻게 구현할지가 이 프로젝트의 첫 숙제였습니다.

- 처음엔 연출용 더미 카드와 포트폴리오 카드를 따로 두고 겹쳐서 페이드로 교차하려 했으나, 경계가 깔끔하게 떨어지지 않아 폐기.
- 앞쪽 연출 카드가 곧 포트폴리오 첫 N개여야 한다는 결론에 이르러, 데이터 조회를 목록 컴포넌트가 아니라 페이지(상위) 레벨로 끌어올려 히어로와 목록이 한 소스를 공유.
- 목록은 히어로가 이미 보여준 앞쪽 N개를 잘라내고 그다음부터 렌더해, 같은 데이터를 두 번 노출하지 않도록 분할.
- 히어로 카드 수와 잘라내는 길이를 같은 반응형 분기(데스크톱 5·태블릿 3·모바일 1)로 묶어 둘을 항상 동기화.
- 단일 스크롤 진행도 하나로 비디오·스켈레톤·실데이터 카드를 시점차를 두고 전환했습니다. 별도 타임라인 도구 없이 연속 시퀀스로 오케스트레이션하되, 대신 수십 개 변형 값이 흩어지는 유지보수 비용은 감수했습니다.

### 호버 시 단일 인스턴스 지연 로딩으로 미디어 비용 최소화

포트폴리오 카드의 클립 영상을, 모든 카드가 미리 들고 있지 않고 호버하는 순간 하나만 불러오도록 설계했습니다. 백엔드도 서버 관리자도 없이 서버 운영을 직접 하는 환경이라, 수십 개 영상을 페이지 진입마다 한꺼번에 받는 구조는 초기 로드와 대역폭을 불필요하게 키우는 비용이라고 봤기 때문입니다.

- 그리드 카드는 정적 썸네일만 렌더하고, 클립 영상 요소는 전역에 하나만 둔 팝오버 안에서 호버 시점에만 마운트.
- 마우스가 스쳐 지나가는 카드는 무시하고 잠깐 머문 카드만 지연 후 로드해, 의도 없는 호버까지 영상을 받지 않도록 거름.
- 화면에 카드가 수십 개여도 그 순간 네트워크를 타는 영상은 머문 카드 하나뿐이라, 초기 로드와 대역폭을 그만큼 절감.
- 팝오버를 카드마다 두지 않고 전역 단일 인스턴스로 둔 덕에, 영상 요소의 생성·해제를 한 곳에서 통제.
- 대가는 호버 첫 순간의 지연입니다. 영상이 즉시 뜨지 않고 마운트 후 재생되지만, 깔려 있던 썸네일이 그 공백을 가린다고 판단했습니다.

### 필터·스크롤 위치를 URL로 목록·상세 왕복 보존

포트폴리오 필터를 클라이언트 상태가 아니라 URL에 두고, 목록에서 상세로 갔다 돌아올 때 보던 위치까지 복원되게 설계했습니다. 필터를 화면 상태로만 들고 있으면 공유·뒤로가기·새로고침에서 모두 깨지고, 상세를 보고 목록으로 오면 맨 위로 튀는 흔한 불편을 없애고 싶었습니다.

- 카테고리 선택을 URL 쿼리로 표현했습니다. 클릭이 곧 주소 이동이고, 서버가 그 값으로 데이터를 다시 조회해 목록을 갱신.
- 필터를 클라이언트가 들고 다니지 않으므로, 특정 카테고리 화면을 그대로 링크로 공유하거나 뒤로가기로 되짚을 수 있음.
- 상세로 들어갈 때 현재 카테고리를 함께 실어 보내고, 돌아올 때 그 카테고리와 보던 카드 위치(앵커)로 복귀해 필터와 스크롤을 URL만으로 왕복.
- 필터 전환마다 서버를 다시 거치는 비용은 있으나, 로딩 스켈레톤으로 그 사이를 메워 흐름을 끊지 않음.

### 온라인 문의: 클라이언트·서버 이중 검증부터 메일 발송까지 한 흐름

온라인 문의를 화면 검증에서 그치지 않고, 서버에서 다시 검증한 뒤 접수와 메일 발송까지 한 흐름으로 끝맺었습니다. 폼 한 장이라도 사용자가 보는 결과를 끝까지 책임지려면 클라이언트 검증만으로는 부족하다고 봤습니다.

- 입력값 검증을 화면(React Hook Form)과 서버(Server Action)에서 이중으로 두어, 클라이언트를 우회한 값도 서버에서 한 번 더 거름.
- 접수(DB 저장)와 알림(메일)을 분리해, 메일 처리가 접수 흐름을 잡지 않게 구성.
- 첨부파일의 한글 파일명이 전송 구간에서 깨지던 문제를, 인코딩으로 감싸 보내고 받는 쪽에서 되돌리는 방식으로 해결.

## 배포와 운영

회사가 직접 쓰는 자사 사이트라 배포와 서버 운영까지 맡았습니다. 영상 작업물 등록과 문의 확인 같은 관리 기능은 별도 사내 백오피스(RGB Managers)에서 담당하고, 이 사이트는 그 데이터를 받아 전시하는 공개 프론트 역할을 합니다.

2024년 오픈 후 회사의 대표 사이트로 지금까지 운영되고 있고, 직접 유지보수를 이어 오고 있습니다.
