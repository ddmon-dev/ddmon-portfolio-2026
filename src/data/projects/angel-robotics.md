## Overview

엔젤로보틱스 홈페이지는 웨어러블 로봇 기업의 기존 사이트를 Next.js 기반 다국어 브랜드 사이트와 운영자 CMS로 리뉴얼한 프로젝트입니다. 저는 PC 시안만 제공받은 상태에서 공개 사이트의 인터랙션 FE와 반응형 구조, 콘텐츠 운영 어드민을 단독으로 구현했습니다.

이 프로젝트에서 내세울 지점은 “화려한 스크롤 효과” 자체가 아닙니다. 웹과 디바이스를 고려하지 않은 시각 흐름을 실제 브라우저에서 성립하는 구조로 다시 저작하고, 그 뒤에서 운영자가 콘텐츠를 계속 관리할 수 있는 CMS까지 붙인 점입니다.

## 작업범위

- 다국어 공개 사이트와 운영자 CMS 개발
- PC 시안 기반 모바일·태블릿 반응형 직접 해석
- 제품 소개 스크롤 인터랙션, 라우트별 스크롤 엔진 적용
- ko/en 콘텐츠 구조와 CMS 관리 화면 구현
- 문의·채용지원 폼, 파일 업로드, 알림 메일 설정 구현
- Supabase 연동과 배포

## 기획 의도

기존 사이트 리뉴얼이었지만, 출발점에 있던 것은 웹 환경을 충분히 고려하지 않은 PC 시안이었습니다. 모바일·태블릿 화면, 스크롤 씬의 동작 방식, 운영 도구는 정의되어 있지 않았기 때문에 단순히 시안을 옮기는 일이 아니라 웹에서 성립하는 흐름으로 다시 구성해야 했습니다.

동시에 콘텐츠가 잦게 바뀌는 기업 사이트였기 때문에, 공개 사이트와 함께 운영자 CMS를 구축했습니다. 이 CMS는 RGB Managers V2에서 정리한 관리 화면 구조를 거래처 프로젝트에 이식한 사례입니다.

## Key Features

- 제품·브랜드 스크롤 인터랙션과 씬 전환
- ko/en 다국어 회사·제품·IR·채용·고객지원 페이지
- 뉴스룸, IR 공시, 채용공고, FAQ, 자료실 등 CMS
- 문의·채용지원 폼의 파일 첨부와 동의 검증
- 운영자가 관리하는 메일 수신자·팝업·게시판 설정

## 핵심 설계 결정

### 디자인 의도를 디바이스별 스크롤 씬으로 재저작

PC 시안이 전제한 연속 스크롤 연출은 좁은 세로 화면에서 그대로 성립하지 않았습니다. 같은 콘텐츠를 디바이스별로 다른 구조로 다시 만들었습니다.

- PC는 sticky 기반 단일 스크롤 씬으로 중앙 비주얼과 설명 콘텐츠가 연속 전환.
- 모바일·태블릿은 같은 내용을 분리된 두 섹션으로 재구성.
- 단순 리플로우가 아니라 구현 두 벌을 유지하는 비용을 감수.
- 스크롤 진행도 임계값을 소수점 단위로 조정하며 실제 동작 기준으로 튜닝.

### 라우트 그룹으로 스무스 스크롤 적용 범위를 선언

스무스 스크롤을 전역에 걸지 않고, 페이지 성격에 따라 라우트 그룹으로 적용 범위를 나눴습니다.

- 제품·브랜드 서사 페이지는 Lenis 스무스 스크롤 그룹에 배치.
- 게시판, 문의, IR처럼 긴 목록과 입력이 있는 데이터 페이지는 네이티브 스크롤 유지.
- 새 페이지는 어느 그룹에 두는지만으로 스크롤 동작이 결정.
- 인터랙션 몰입감과 데이터 탐색성의 균형을 라우트 구조로 표현.

### 다국어 카피는 번역 파일보다 화면 옆에 둠

정적 카피는 별도 번역 카탈로그보다 컴포넌트 근처에 두는 방식을 택했습니다. 에이전시 작업에서는 문구 수정과 개행 조정이 잦아, 실제 마크업 위치를 보며 바로 고치는 편이 비용이 낮았습니다.

- `content[lang]` 객체를 현재 언어로 인덱싱해 문자열·JSX·반응형 개행을 함께 관리.
- 디자인 의도에 맞춘 줄바꿈과 강조 표현을 화면 단위로 조정.
- 키 누락을 컴파일 단계에서 잡기 어려운 비용은 2개 언어 규모에서 수용.
- DB 콘텐츠는 언어별 테이블 분리로 운영 데이터와 정적 카피를 분리.

### 사내 백오피스 구조를 거래처 CMS로 이식

18종 안팎의 콘텐츠 관리 화면을 처음부터 새로 짜지 않고, RGB Managers V2에서 만든 목록·검색·작성/수정 구조를 이식했습니다.

- 목록, 검색, 페이지네이션, 생성/수정 패널을 같은 관리 화면 패턴으로 구성.
- 화면마다 다른 것은 대상 데이터와 필드 정의로 제한.
- 정교한 제네릭 프레임워크라기보다, 이미 검증한 사내 구조를 거래처 운영 도구에 빠르게 적용한 레버리지.
- 콘텐츠 운영까지 포함한 브랜드 사이트 납품 흐름을 만들었다는 점이 핵심.

### 파일 첨부 폼 검증을 선언적 스키마로 묶음

문의와 채용지원 폼은 파일 크기, 확장자, 필수 동의 여부가 얽혀 있어 단순 입력 폼으로 처리하기 어려웠습니다.

- 파일 크기·확장자·필수 개수를 필드 정의에서 검증 스키마로 생성.
- 이력서 첨부와 개인정보 동의 같은 도메인 규칙을 입력 단계에서 강제.
- 브라우저에서 스토리지로 직접 업로드한 뒤 서버는 결과를 검증·저장.
- 플랫폼 요청 본문 제한을 피하면서 서버 부하도 줄인 구조.

## 트러블슈팅 · 운영 한계 대응

### 스무스 스크롤과 모달 스크롤 잠금 충돌

스무스 스크롤이 켜진 페이지에서 모달을 열면 본문 스크롤 제어와 모달 내부 스크롤이 충돌했습니다. 모달이 열릴 때 본문 스크롤만 멈추고 내부 스크롤 영역은 통과시키도록 게이팅해, 인터랙션 페이지에서도 폼·모달을 사용할 수 있게 조정했습니다.

### 요청 본문 크기 제한을 클라이언트 직접 업로드로 우회

파일 첨부를 서버 액션으로 받던 구조가 배포 플랫폼의 요청 크기 제한에 막혔습니다. 파일은 브라우저에서 스토리지로 직접 올리고, 서버는 업로드 결과와 폼 데이터만 검증하는 경로로 바꿔 제한을 우회했습니다.

## 배포와 운영

알지비커뮤니케이션즈 재직 중 공개 사이트와 어드민을 단독으로 구축해 오픈했고, 현재 angel-robotics.com으로 운영 중입니다. 이 프로젝트는 운영 지속보다, 디자인 의도와 웹 구현 사이의 간극을 프론트엔드 구조로 메운 사례로 두는 것이 적합합니다.
