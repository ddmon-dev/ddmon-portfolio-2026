## Overview

웹을 전제하지 않은 브랜드 디자인을, 디바이스마다 성립하는 스크롤 인터랙션 구조로 다시 설계한 프로젝트입니다. 웨어러블 로봇 기업의 기존 홈페이지를 한국어·영어 다국어 사이트로 리뉴얼하면서, 공개 사이트의 인터랙션 FE와 운영자용 어드민 CMS를 디자인을 제외하고 단독으로 맡았고, 현재까지 운영 중입니다.

공개 사이트는 제품·IR 공시·채용·고객지원에 걸친 29개 화면, 어드민은 콘텐츠를 ko/en으로 관리하는 CMS입니다. 어드민은 매번 새로 짜는 대신 직접 만들어 운영하던 사내 시스템(RGB Managers)의 구조를 이식해 빠르게 세웠습니다. 검증된 자산을 거래처 제품에 재사용한 레버리지입니다.

## 작업범위

- 1인 단독 개발, 공개 사이트·어드민 2개 앱(공개 29개 화면 + 어드민 콘텐츠 관리 약 18종)
- 디자인은 PC 홈페이지 시안만 제공, 모바일·태블릿 반응형과 어드민 화면은 직접 해석·설계
- 디자이너 1명·3D 영상작업자 1명과 협업
- 스크롤 인터랙션 중심 FE 구현과 ko/en 다국어 구조
- 화면을 끝까지 책임지기 위해 어드민 CMS·Supabase·배포까지 내려간 범위

## 기획 의도

기존 사이트를 현대 스택으로 다시 짓는 리뉴얼이었고, 시작점에 있던 건 웹·스크롤 환경을 전제하지 않은 PC 홈페이지 시안 하나뿐이었습니다. 인터랙션도 모바일·태블릿 화면도 운영 도구도 정의되지 않은 상태였습니다. 그래서 제 몫은 시안을 옮기는 일이 아니라, 디자인이 의도한 흐름을 실제 웹에서 깨지지 않는 구조로 다시 설계하는 일이었습니다.

동시에 콘텐츠를 운영자가 코드 배포 없이 직접 관리하도록 CMS를 함께 세웠고, 이건 검증된 사내 시스템 구조를 이식해 빠르게 올렸습니다. 일회성 납품이 아니라 다음 프로젝트에도 재사용할 자산으로 접근한 부분입니다.

## Key Features

- 제품·브랜드 스크롤 인터랙션: Lenis 스무스 스크롤 + 스크롤 연동 씬 전환
- ko/en 다국어 공개 사이트(정적 생성)와 9종 게시판(뉴스룸·IR 공시·채용공고·FAQ·자료실 등)
- 파일 첨부·다중 동의 검증이 들어간 문의·채용지원 폼, 병원찾기, 자료 다운로드, 팝업 관리
- 콘텐츠를 다국어로 관리하는 어드민 CMS와, 운영자가 무배포로 다루는 메일 설정

## 핵심 설계 결정

### 디바이스별 스크롤 씬 재저작으로 디자인 의도 보존

PC 시안이 전제한 "한 요소가 끊기지 않고 이어지는" 연출을, 디바이스마다 성립하는 별개 구조로 다시 저작했습니다. 대표적으로 제품(M20) 어플리케이션 소개 섹션은 좁은 세로 화면에서 그 연속성이 깨졌기 때문입니다.

- PC는 단일 sticky 스크롤 씬. 스크롤 진행도를 씬 번호로 매핑해, 중앙 비주얼이 연속 변형되는 동안 설명 콘텐츠가 교차 전환.
- 태블릿·모바일은 같은 섹션을 분리된 두 섹션으로 재구성했습니다. 연속 연출이 좁은 세로 뷰포트에서 깨지므로 구조 자체를 다시 저작.
- 디바이스 분기로 같은 콘텐츠에 구현 두 벌의 유지 비용을 감수했습니다. 단순 리플로우로는 디자인 의도가 살지 않는다는 판단.
- 씬이 바뀌는 임계값은 스크롤 진행도 기반으로 소수점 단위 5단계까지 수작업 튜닝.

### 라우트 그룹으로 스무스 스크롤 적용 범위 선언

스무스 스크롤을 전역에 거는 대신, 라우트 그룹 단위로 켜지고 꺼지도록 선언했습니다. 라이브러리를 어디 켜고 끌지를 코드 분기가 아니라 라우트 구조로 표현한 결정입니다.

- 콘텐츠·제품 페이지는 스무스 스크롤 그룹, 게시판·문의·IR 같은 데이터 페이지는 네이티브 스크롤 그룹으로 분리.
- 긴 동적 리스트에서는 관성 스크롤이 오히려 탐색을 방해한다고 보고, 데이터 화면은 의도적으로 제외했습니다.
- 새 페이지는 어느 그룹에 두느냐만으로 스크롤 동작이 결정되어, 페이지별 분기 코드 없이 일관.

### 사내 시스템(RGB Managers) 구조 이식으로 18종 CMS 구축

18종 콘텐츠 관리 화면을 매번 새로 짜는 대신, 직접 만들어 운영하던 사내 시스템의 구조를 이식했습니다. UI·동작은 공통 한 벌로 묶고, 화면마다 다른 건 대상 데이터만 갈아끼우는 방식입니다.

- 목록·검색·페이지네이션·생성/수정 패널은 공통 컴포넌트 한 벌로, 라우트가 대상 테이블만 주입하면 동작.
- 헤드리스 테이블(@tanstack/react-table)을 도메인 비의존 표현 컴포넌트로 감싸 정렬·빈 상태·행-편집 패널 연동을 흡수.
- 치른 비용: UI·동작은 공통이어도 데이터 접근 코드는 모듈마다 격리돼, 검증 정책을 바꾸면 여러 모듈을 함께 손봐야 하는 중복을 안고 감.

### i18n 라이브러리 없이 콜로케이션 + 언어별 테이블 다국어

국·영문을 i18n 라이브러리 없이, 번역을 마크업 옆에 두는 콜로케이션 방식으로 처리했습니다. 에이전시 작업 특성상 수정이 잦고 개행 등 디자인 구현 정밀도가 중요해, 번역을 한곳에 모으기보다 화면에서 위치를 보고 바로 고치는 편이 변경 비용이 낮다는 판단이었습니다.

- 정적 카피는 `content[lang]` 객체를 현재 언어로 인덱싱해, 값에 문자열·JSX·반응형 개행까지 담아 디자인 의도대로 표현.
- DB 콘텐츠는 언어별 테이블 분리(`newsroom_ko` / `newsroom_en`), 라우트의 언어 파라미터를 테이블 접미사로 사상해 모듈 수가 (모듈 × 언어)로 불어나지 않음.
- 치른 비용: 번역 키 누락을 컴파일 단계에서 못 잡고, 복수형·날짜 포맷(ICU) 기능을 포기했습니다. 2개 언어·이 규모에서 수용 가능한 대가.

### URL 단일 원천 목록 상태로 공개·어드민 일관성 확보

게시판·관리 화면의 검색·페이지·필터를 React 상태가 아니라 URL을 단일 진실원으로 삼아 다뤘습니다. 새로고침·뒤로가기·링크 공유가 그대로 동작하게 만들기 위해서입니다.

- 검색어·페이지·카테고리가 전부 URL에 직렬화돼, 공개 사이트와 어드민이 같은 방식을 공유.
- 블록 페이지네이션을 라이브러리 없이 직접 구현하고, 그 계산 유틸을 화면 표시와 DB 조회가 함께 사용.
- 치른 비용: 필터 변경이 서버 내비게이션을 타므로 즉각적인 클라이언트 인터랙션은 포기.

### 선언적 스키마로 파일 첨부 폼 검증 동적 생성

채용지원·문의 폼의 파일 검증을, 컴포넌트마다 따로 짜는 대신 필드 정의만 받아 검증 규칙을 생성하는 선언적 스키마로 묶었습니다.

- 파일 크기·확장자·필수 개수·배열 여부를 선언으로 받아 검증 스키마를 동적 생성하므로, 새 파일 폼은 정의만 추가.
- 이력서 첨부(용량·문서 확장자·최소 1개), 개인정보 필수/선택 동의 같은 도메인 규칙을 입력 단계에서 강제.
- 클라이언트 검증과 별개로 서버에서 다시 검증해, 표시는 클라이언트, 권위는 서버로 분리.

## 트러블슈팅 · 운영 한계 대응

### 스무스 스크롤과 모달 스크롤 잠금의 경합을 직접 게이팅으로 해소

스무스 스크롤을 켠 페이지에서 모달을 열면, 모달의 스크롤 잠금과 스크롤 라이브러리의 제어가 충돌했습니다. 둘이 같은 스크롤을 두고 경합하는 것이 원인이라 진단하고, 모달이 열리면 본문 스크롤만 멈추되 모달 내부 스크롤 영역은 통과시키도록 직접 게이팅해 풀었습니다.

### 플랫폼 요청 본문 크기 제한을 클라이언트 직접 업로드로 우회

파일 첨부를 서버 액션으로 받던 구조가 배포 플랫폼의 요청 본문 크기 제한에 막혔습니다. 원인을 서버 경유 업로드로 진단하고, 브라우저에서 스토리지로 직접 업로드한 뒤 그 결과만 서버가 검증·저장하도록 경로를 바꿔 풀었습니다. 서버 부하도 함께 덜어낸 전환입니다.

## 배포와 운영

알지비커뮤니케이션즈 재직 중 공개 사이트와 어드민, 배포 환경까지 단독으로 구성해 오픈했습니다. 재직 중 운영을 안정화하는 과정에서 실데이터 입력의 다양성에 맞춰 전화번호 입력 포맷터를 대표번호·지역번호 등 다중 패턴까지 보강했고, 폼 알림 수신자를 한 명에서 여러 명으로 확장하면서 기존 데이터를 무중단으로 호환 전환했습니다.

웨어러블 로봇 브랜드의 다국어 공개 사이트로 현재까지 운영되고 있습니다. 직접 만든 사내 시스템 구조를 거래처 제품에 이식해 다국어 CMS를 빠르게 세운, 재사용 자산이 실제 납품으로 이어진 사례입니다.
