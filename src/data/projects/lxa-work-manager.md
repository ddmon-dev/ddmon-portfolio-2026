## Overview

외주 고객·견적·프로젝트·세무를 하나의 데이터 흐름으로 묶어, 직접 정의하고 설계해 운영 중인 본인 IP 업무 도구입니다. 제가 유일한 사용자이자 기획자·디자이너라, 무엇을 만들지부터 업무를 어떤 데이터 구조로 옮길지까지 제품 전체를 직접 판단했습니다. 일이 들어올 때마다 견적을 뽑고 프로젝트와 세무를 처리하는 데 매일 쓰고 있습니다.

Next.js + Supabase 기반 단일 앱으로 빠르게 만든 초기 버전을, 한 달이 안 돼 React SPA + Hono + Drizzle로 전면 재작성했습니다. 코드 대부분은 AI로 작성했고, 제 몫은 아키텍처를 어떻게 가를지, 업무를 어떤 테이블로 옮길지, 화면이 어떻게 동작할지를 판단하고 그 결과를 검증·디렉션하는 일이었습니다. 그래서 이 프로젝트는 손으로 짠 코드의 깊이가 아니라, AI가 코드 쓰는 비용을 거의 0으로 만든 뒤 남는 진짜 병목인 기획·설계·판단에 시간을 쏟은 기록입니다.

## 작업범위

- 단독 개발 · 본인 IP, 레거시와 전면 재작성 두 버전
- FE 화면·UX·인터랙션 설계, 디자인 시안 없이 직접 해석
- 업무 도메인·데이터 모델링과 시스템 아키텍처 방향 결정(단일 앱 → 레이어 분리)
- 코드 구현은 AI 드리븐, 경험이 없던 백엔드·모노레포는 AI에 위임하고 구조를 파악·디렉션
- self-hosted 로컬 상시 구동·일일 백업 환경 운영
- 자기 도구를 자연어로 운영하는 AI 스킬 레이어 실험

## 기획 의도

알지비 사내 운영 시스템(RGB Managers)을 직접 만들어 본 경험의 연장이었습니다. 남의 회사 업무를 시스템으로 묶어본 뒤, 정작 제 프리랜스 업무(고객 관리, 견적, 프로젝트 진행, 부가세·원천징수 세무)는 흩어진 채라는 게 눈에 들어왔습니다. 내 업무 흐름이니 요구사항을 누구보다 정확히 안다는 점에서 직접 만들기로 했습니다. 기획·디자인·개발의 경계 없이 "무엇을 만들지"를 스스로 정의해야 했고, 그게 이 프로젝트 내내 가장 큰 일이었습니다.

어드민 골격(목록·검색·작성/수정)은 본인 IP인 어드민 템플릿에서 이식해 출발점으로 삼았습니다. 덕분에 반복되는 CRUD 화면은 빠르게 세우고, 머리는 견적·프로젝트 같은 도메인 고유의 구조에 쏟을 수 있었습니다.

## Key Features

- 고객 관리: 연락처·메모·환경 정보 CRM
- 견적서: 시리즈/버전 관리와 최종 버전 지정, 부가세·원천징수 산식, PDF 출력과 라이브 미리보기
- 프로젝트: 견적 연계, 청구·수금·원가·외주지급 4개 장부, 세금계산서·원천세 정산
- 매출·정산 통계, 일정·간트·투두·메모 위젯
- Claude Code 스킬로 자연어 운영: 기존 견적 데이터 기반 견적 산출 자동화 시도

## 핵심 설계 결정

### 단일 사용자 전제로 Supabase 단일 앱 → 자체 4레이어 재작성

잘 돌아가던 Next.js + Supabase 단일 앱을, 한 달이 안 돼 React SPA + Hono + Drizzle + Better Auth 네 레이어로 분해해 재작성했습니다. 기능이 부족해서가 아니라, 이 도구의 운영 맥락에 스택이 과하다고 판단했기 때문입니다.

- 사용자가 한 명인 도구라 웹서버로 띄울 이유가 없고 로컬에서 돌면 충분한데, 그 로컬에 여러 컨테이너로 뜨는 Supabase 스택을 상시 구동하는 건 무거웠음.
- Supabase를 인증·실시간 없이 DB·스토리지로만 쓰고 있었음. BaaS의 값을 안 내는 셈이라, 자체 백엔드·DB가 운영 자유도가 더 높다고 판단.
- 유지보수 예지: 레거시는 다중 테이블 트랜잭션(견적=시리즈+버전+그룹+항목)을 SDK 한계상 plpgsql RPC로 처리했는데, 기능이 늘수록 비즈니스 로직이 DB 함수로 흩어질 흐름이었음. 코드 레벨 트랜잭션으로 로직을 앱 한 곳에 모으는 쪽을 택함.
- 치른 비용: 서버 로직 전부를 REST로 다시 쓰고 모노레포 두 앱을 운영하는 부담. 1인 도구의 수명·유지보수를 보면 남는 거래라고 보고 택한 트레이드오프.

### 견적 워크플로우를 깨지지 않는 테이블 불변식으로 모델링

견적이 실무에서 어떻게 흐르고 어디서 어긋나는지를 먼저 분해한 뒤, 그것을 깨지지 않는 테이블 불변식으로 옮겼습니다. 이 프로젝트에서 가장 큰 일은 코드가 아니라 이 데이터 구조 설계였습니다.

- 견적을 시리즈 → 버전 → 그룹 → 항목 4계층으로 정규화해, 같은 견적의 개정 이력을 버전으로 쌓고, "어느 버전이 계약본인지"는 별도 플래그 없이 최종 버전 포인터 하나로 지정.
- "계약 상태 ⟺ 최종 버전 지정"을 DB 제약으로 강제해, 애플리케이션 버그가 데이터 정합을 깨지 못하도록 불변식을 스키마에 박음.
- 프로젝트의 돈 흐름을 청구·수금·원가·외주지급 4개 장부로 분리하고, 부가세(별도 청구)와 원천징수(3.3% 차감)를 회계 산식으로 분기해, 한국 프리랜스 세무의 현실을 데이터 모델에 녹임.
- 처음부터 완벽히 맞히지 못해, 수금 모델을 단일 테이블 → 세금계산서 분리 → 통합 수금으로 점진적으로 리팩토링. 실무를 쓰며 드러난 엣지케이스를 모델에 되돌린 흔적.

### 검증·금액 계산을 모노레포 공유 패키지로 단일화 (프론트·서버·PDF)

화면이 약속한 금액과 서버가 저장하는 금액, PDF에 찍히는 금액이 어긋나지 않도록, 검증 규칙과 세금·금액 계산을 모노레포 공유 패키지로 단일화했습니다.

- 같은 Zod 스키마가 프론트 폼 검증과 서버 입력 검증을 동시에 구동해, 폼에서 막힌 입력은 서버에서도 같은 규칙으로 막힘.
- 부가세·원천징수 계산 커널을 프론트 폼·상세·PDF·서버가 공유하므로, 세법이나 산식이 바뀌어도 고칠 곳은 한 군데.
- 검증 스키마는 DB에서 자동 생성하지 않고 손으로 작성. 한국어 검증 메시지와 폼 전용 규칙을 직접 통제하려는 선택입니다. 자동화된 단일소스가 아니라, 공유하는 검증·계산 커널의 단일화.
- 서버는 클라이언트가 보낸 금액을 믿지 않고 다시 계산. 금액을 다루는 시스템의 신뢰 경계를 서버에 둠.
- 트레이드오프: 프레임워크가 제공하는 자동 타입 공유(RPC) 대신 공유 패키지 + 명시적 응답 타입을 택해, 마법보다 추적 가능한 경계를 우선.

### AI 드리븐 구현, 기획·설계·검증·디렉션은 본인 몫

이 도구는 코드 대부분을 AI로 작성했습니다. 그래서 제 역할을 "직접 짠다"가 아니라 "무엇을 만들지 정하고, 나온 결과를 검증·디렉션한다"로 설계했고, 그게 AI 시대에 제가 일하는 방식입니다.

- 어려움의 무게추가 옮겨갔음. 코드 작성은 빠르게 흘렀고, 정작 시간이 든 건 기획·데이터 설계·UI/디자인 같은 판단의 영역.
- 디자인은 원하는 방향이 있어도 그걸 텍스트로 AI에 전달하고 핀포인트로 교정하는 게 가장 까다로웠음. 감각을 언어로 옮기는 마찰을 직접 겪으며 다룸.
- 경험이 없던 백엔드·모노레포 구성은 AI에 위임하되, 어떤 구조인지 끝까지 파악해 두고 디렉션. 모르는 채로 맡기지 않음.
- 분해로 생긴 REST 경계가 브라우저 UI 밖에서 같은 데이터를 재사용하는 길을 열어, 자연어로 도구를 운영하는 스킬 레이어를 실험. 위험한 동작(금액 변경·삭제·상태 전환)은 미리보기 후 동의 게이트로 막고, 평가 케이스로 의도대로 동작하는지 점검(점검 중 스킬 문서와 실제 API가 어긋난 결함 2건을 포착·수정).
- 다만 핵심 목표였던 견적 산출 자동화는 아직 만족스러운 완성도에 이르진 못한 진행 중 실험. 도구를 자연어로 모는 방식의 가능성과 한계를 함께 확인.

## 배포와 운영

웹서버로 띄울 필요가 없는 1인 도구라, 제 머신에서 상시 구동되는 로컬 서비스로 운영합니다. 서버가 빌드된 프론트까지 함께 서빙하도록 묶고, 개발/운영 환경을 분리하고, 매일 DB를 자동 백업하도록 구성했습니다. 규모를 키우는 인프라가 아니라, 1인 도구에 필요한 만큼만 갖춘 운영입니다.

레거시를 만든 직후 전면 재작성에 들어갔지만 데이터 이전이나 운영에서 골치 아픈 일은 없었고, 지금은 일이 들어올 때마다 견적·프로젝트·세무를 이 도구로 처리합니다. 트래픽도 협업도 없는 작은 시스템이지만, 내 업무 흐름이 바뀔 때마다 직접 고쳐 쓰며 계속 키워왔습니다. 만들고 끝낸 게 아니라 실제로 쓰는 도구입니다.
