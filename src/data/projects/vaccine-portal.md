## Overview

백신 연구개발·상용화를 지원하는 공공 사업단의 정보 포털입니다. 흩어져 있던 지원시설 현황과 행사·교육 정보를, 사용자가 카테고리별 조건으로 좁혀 찾는 검색 포털과 운영자가 데이터를 직접 관리하는 CMS로 구성했습니다.

2024년 8월 오픈해 현재까지 공공 사업단이 직접 운영을 이어가는 서비스입니다.

## 작업범위

- 사용자 포털·운영자 CMS 두 앱의 화면 설계와 FE 개발
- PC 디자인 시안만 받아 모바일·태블릿 반응형은 직접 해석·구성
- 사용자 포털의 복합 조건 검색 UI·컴포넌트 설계
- 카테고리별 한글 문서로만 있던 원자료를 검색 가능한 데이터 구조로 설계·적재
- 백엔드 담당자 없이 DB 설계부터 셀프호스팅 배포·운영까지

## 기획 의도

흩어져 있던 백신 지원시설 현황을 누구나 조건으로 찾을 수 있는 공개 포털로 만드는 프로젝트였습니다. 정작 원자료는 카테고리별 한글 문서로만 존재해 그대로는 검색이 불가능했고, 그래서 이 프로젝트의 본질을 "비정형 문서를 조건으로 탐색 가능한 데이터로 바꾸는 일"로 정의하고 데이터 구조와 검색 경험 설계에 무게를 뒀습니다.

일정은 파노라마 필름과 겹친 한 달이었습니다. 파노라마에서 만든 공통 구조와 컴포넌트를 그대로 이식해 반복 작업을 걷어내고, 남는 시간을 이 프로젝트 고유의 문제인 데이터 모델과 검색에 집중하는 전략을 택했습니다.

## Key Features

- 시설 카테고리(비임상·임상·위탁생산)별 복합 조건 검색·필터
- 카테고리마다 다른 상세 속성(생물안전등급·시험역량·인증·GMP 등) 탐색
- 행사·교육 일정 캘린더, 게시판
- 운영자 CMS: 시설·행사·교육·게시판·팝업·검색키워드 관리, 구독자 명단 엑셀 추출

## 핵심 설계 결정

### 검색 결과를 공유 단위로: 필터 상태를 URL에 직렬화

카테고리별로 4~9개에 달하는 검색 필터를, 검색 결과 자체가 공유·재방문의 단위라고 보고 필터 상태를 URL에 직렬화했습니다. 검색이 핵심인 포털에서는 "어떤 조건으로 좁힌 결과인지"가 링크 한 줄로 전달돼야 한다고 봤습니다.

- 필터·검색어를 URL에 담아 결과 링크 공유·북마크·뒤로가기가 자연스럽게 동작.
- 체크박스 변경은 임시 상태에만 모으고 검색 실행 시 한 번에 URL로 커밋해, 필터가 많은 화면에서 입력 중 매 토글마다 서버 왕복·리렌더가 일어나지 않게.
- 다중 선택값은 단일 쿼리키에 직렬화하고, URL 변화는 다시 필터 UI로 역동기화해 직접 진입·뒤로가기에도 상태 복원.
- 임시 상태와 확정 상태를 가르는 양방향 동기화 복잡도를 치르는 대신, 공유 가능성과 새로고침 안전을 택한 선택.

### 좁은 화면 대응: 비교 표를 뷰포트별 역량 태그 카드로 재구성

생물안전등급·시험역량 등 13개 열의 시설 비교 표를, 좁은 화면에 그대로 옮기면 정보 밀도가 무너진다고 보고 모바일에서는 같은 데이터를 역량 태그 카드로 다르게 렌더했습니다. PC 시안만 받은 상태라 모바일 경험은 직접 해석해야 했습니다.

- 같은 데이터를 뷰포트별로 다르게 표현해, 데스크탑은 rowSpan/colSpan 비교 표, 모바일은 보유 역량만 추린 해시태그 카드.
- 셀의 보유 여부(해당 시험·인증)를 모바일에서는 태그로 환원해 스캔성 확보.
- 표와 카드 두 마크업을 동시에 두는 비용을 치르는 대신, 화면 폭에 맞는 정보 밀도를 택함.

### 필터 축이 곧 쿼리가 되도록 비정형 문서를 검색 가능 데이터로 설계

화면의 필터 축을 먼저 정하고, 사용자가 누른 조건이 곧 데이터베이스 조회가 되도록 데이터 구조를 거기에 맞춰 설계했습니다. 카테고리별 한글 문서로만 있던 원자료를 검색 가능한 형태로 재정의한 것이 이 프로젝트의 본질입니다.

- 같은 시설이 여러 카테고리 문서에 중복 등장하므로, 공통 시설정보를 중심 테이블로 묶고 카테고리별 상세를 분리 테이블로 외래키 연결(공통 속성 정규화).
- 카테고리마다 형태가 다른 역량 매트릭스(독성·효력시험, GMP·CDMO 단계 등)는 JSONB로 흡수해, 카테고리가 늘어도 스키마가 폭발하지 않도록 의도적 비정규화.
- 한 시설의 다중 카테고리 소속은 멤버십 플래그로 표현해 필요한 카테고리만 조회.
- 단일 거대 테이블은 열 폭주로, 카테고리를 잘게 쪼갠 과한 조인 구조는 쿼리 부담으로 각각 피하고, 카테고리 수가 고정적이라는 판단으로 분리 테이블 추가 비용을 수용.

### 캐시보다 신선도: stale 데이터를 운영 비용으로 본 조회 전략

운영자가 CMS에서 고친 내용이 사용자 화면에 곧바로 반영되는 것을 기준으로 삼고, 시설 데이터가 대량이 아니라는 점을 이용해 조회 전략을 거기에 맞췄습니다. 정보가 어긋나 보이면 곧 문의·수정 요청이라는 운영 비용이 되므로, 캐시보다 신선도를 우선했습니다.

- 필터 옵션(지역·국가)·시설 카운트는 전체 집합이 있어야 산출되고 여러 페이지가 공유하므로, 진입 시 한 번 로드해 세션 내 재조회 없이 재사용.
- 검색 결과는 조건 조합이 매번 달라 서버 조회로 처리하고, 무거운 조건 로직은 화면에서 분리.
- 운영자 수정이 즉시 반영되도록 신선도를 우선한 것은, stale 데이터로 인한 문의를 운영 비용으로 본 트레이드오프.

### 이질적 카테고리 속성을 표 형태 + 조건부 동적 폼으로 입력

카테고리마다 평가 축이 전혀 다른 시설 역량 정보를, 운영자가 코드 없이 채울 수 있는 표 형태 입력 폼으로 설계했습니다. 독성·효력시험, GLP 준수, 생물안전등급, GMP·CDMO 단계처럼 구조가 다른 데이터를 일반 게시판으로는 다룰 수 없었습니다.

- 카테고리별 상세 항목을 표 형태 폼으로 모델링해 시험·인증 보유 여부를 항목 단위로 입력.
- "기타" 선택 시 입력 행이 동적으로 늘어나는 조건부 폼으로 비정형 항목까지 수용.
- 수정 시 기존 파일을 보존·교체·삭제로 구분 처리하는 파일 입력으로 운영 중 갱신 흐름 반영.
- 운영자가 코드 없이 시설 역량표를 직접 갱신하는 제품 흐름을 화면으로 설계.

## 트러블슈팅 · 운영 한계 대응

### PostgREST가 표현 못 하는 복합 조건을 DB 1차 필터 + 서버 후처리로

여러 역량을 동시에 만족하는 시설을 찾는 검색에서, JSONB 안의 다중 조건을 PostgREST 쿼리 하나로 표현하기 어려운 한계를 만났습니다. 구조적 필터는 데이터베이스 조회로 후보를 먼저 좁히고, 한 쿼리로 떨어지지 않는 복합 조건은 서버에서 추가로 가르는 방식으로 검색을 완성했습니다.

### 셀프호스팅 운영 중 로그 누적으로 인한 다운

운영 중 사이트가 다운돼 서버를 확인하니, 셀프호스팅 스택의 로그가 디스크를 가득 채워 서비스가 멈춘 상태였습니다. 로그 출력·보관 설정을 조정해 재발을 막았습니다. 매니지드 서비스였다면 마주치지 않았을 문제로, 내가 띄운 화면이 멈추지 않도록 원인을 끝까지 추적해 막은 사례입니다.

## 배포와 운영

원래 Supabase 매니지드와 Vercel로 배포할 계획이었으나, 공공기관 특성상 해외 유료 서비스의 결제 절차가 막혀 배포 직전 전환이 필요했습니다. 화면을 끝까지 띄우기 위해 가비아 클라우드 우분투 서버에 Supabase와 앱을 직접 올리고 Nginx로 묶어, 자료를 찾아가며 배포 환경을 단독으로 구성했습니다.

소스가 외부에 공개되지 않는 납품 서비스이지만, 공개 URL로 지금도 동작을 확인할 수 있고 2024년 8월 오픈 후 공공 사업단이 직접 운영·유지보수를 이어가고 있습니다.
