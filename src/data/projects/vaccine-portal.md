## Overview

백신정보포털은 한글 문서로 흩어져 있던 백신 R&D 지원시설 정보를, 사용자가 조건별로 좁혀 찾을 수 있는 검색 포털과 운영자가 직접 관리하는 CMS로 바꾼 프로젝트입니다. 개발은 파노라마 필름과 병행한 약 한 달 동안 단독으로 진행했습니다.

이 프로젝트에서 중요한 일은 화면을 많이 만든 것이 아니라, 문서 형태의 원자료를 검색 가능한 데이터와 필터 UI로 다시 설계한 점입니다. 사용자가 누르는 필터가 곧 데이터 조회 조건이 되도록, 화면의 탐색 축에서 데이터 모델을 거꾸로 끌어냈습니다.

## 작업범위

- 사용자 검색 포털과 운영자 CMS 개발
- 카테고리별 한글 원자료 정리, 데이터 구조 설계, DB 적재
- 지원시설 복합 조건 검색 UI와 필터 컴포넌트 설계
- PC 시안 기반 모바일·태블릿 반응형 직접 해석
- 관리자 CMS의 시설·행사·교육·게시판 관리 화면 구현
- Supabase·Next 앱 셀프호스팅 배포와 운영 이슈 대응

## 기획 의도

원자료는 카테고리별 한글 문서로 나뉘어 있었고, 같은 시설이 여러 카테고리에 중복 등장했습니다. 그대로 게시판에 옮기면 검색 포털이 아니라 문서 보관함에 가까워지기 때문에, 먼저 사용자가 어떤 조건으로 시설을 찾을지 정의하고 그 필터 축에 맞춰 데이터를 재구성했습니다.

파노라마 필름에서 만든 어드민·브랜드 사이트 구조가 이미 있었기 때문에 반복 화면은 빠르게 이식하고, 남는 시간을 이 프로젝트 고유의 문제인 검색 경험과 데이터 모델링에 집중했습니다.

## Key Features

- 비임상·임상·위탁생산 카테고리별 시설 검색
- 지역, 국가, 생물안전등급, 시험역량, 인증, GMP 등 복합 필터
- 데스크탑 비교 표와 모바일 역량 태그 카드
- 행사·교육 일정, 게시판, 팝업 관리
- 운영자 CMS와 구독자 명단 엑셀 추출

## 핵심 설계 결정

### 검색 결과를 공유 가능한 URL 상태로 설계

검색 포털에서 중요한 것은 어떤 조건으로 좁힌 결과인지입니다. 그래서 필터와 검색어를 컴포넌트 상태가 아니라 URL에 반영해, 결과 자체를 공유·복원 가능한 단위로 만들었습니다.

- 검색어·카테고리·필터 조건을 URL 쿼리로 직렬화.
- 뒤로가기, 새로고침, 직접 진입 시에도 같은 조건 복원.
- 체크박스 변경은 임시 상태에 모으고 검색 실행 시 한 번에 URL로 커밋.
- 필터가 많은 화면에서 입력 중 매 토글마다 서버 이동이 발생하지 않도록 조정.

### PC 비교 표를 모바일 역량 태그 카드로 재저작

13개 안팎의 열을 가진 시설 비교 표는 모바일에 그대로 옮기면 정보 밀도가 무너졌습니다. 같은 데이터를 화면 폭에 따라 다른 표현으로 바꿔 읽히게 했습니다.

- 데스크탑은 rowSpan/colSpan을 활용한 비교 표로 시설 간 차이를 노출.
- 모바일은 보유 역량만 추린 태그 카드로 전환해 스캔성 확보.
- PC 시안만 있는 상태에서 좁은 화면의 정보 구조를 직접 재설계.
- 두 마크업을 유지하는 비용보다 디바이스별 가독성을 우선.

### 필터 축에서 데이터 모델을 역으로 설계

사용자가 선택할 조건이 곧 DB 조회 조건이 되도록, 한글 문서의 시설 정보를 공통 정보와 카테고리별 상세 정보로 나눴습니다.

- 공통 시설정보는 중심 테이블로 묶고, 카테고리별 상세는 별도 테이블로 분리.
- 카테고리마다 다른 시험역량·인증 매트릭스는 JSONB로 흡수.
- 단일 거대 테이블의 열 폭주와 과도한 조인 구조를 모두 피한 절충.
- 카테고리 수가 고정적이라는 도메인 판단 아래 분리 테이블 추가 비용을 수용.

### 캐시보다 신선도를 우선해 운영 문의 비용을 줄임

운영자가 CMS에서 수정한 내용은 사용자 화면에 바로 반영되어야 했습니다. 공공 정보 포털에서 오래된 정보는 곧 문의와 정정 요청이 되기 때문입니다.

- 시설 규모가 크지 않아 옵션·카운트용 참조 데이터는 레이아웃에서 로드.
- 실제 검색 결과는 조건 조합에 따라 서버 조회로 처리.
- 관리자 수정 후 사용자 화면 즉시 반영을 우선.
- 캐싱 이득보다 stale 데이터로 생기는 운영 비용을 더 크게 본 판단.

### PostgREST가 표현하기 어려운 조건은 서버 후처리로 완성

JSONB 안의 여러 역량 조건을 한 쿼리로 모두 표현하기 어려운 지점이 있었습니다.

- 구조적 필터는 DB 조회로 먼저 후보 축소.
- 복합 boolean 조건과 일부 자유 텍스트 조건은 서버에서 추가 필터링.
- 데이터 규모가 제한적인 포털이라는 전제에서 단순성과 구현 안정성을 우선.
- 규모가 커지면 전문 검색이나 별도 인덱싱으로 옮겨갈 수 있는 경계로 인식.

## 트러블슈팅 · 운영 한계 대응

### 공공기관 결제 제약으로 셀프호스팅 전환

처음에는 Supabase와 Vercel을 사용할 계획이었지만, 공공기관 특성상 해외 유료 서비스 결제 절차가 어려워 배포 직전 전환이 필요했습니다. 가비아 클라우드 우분투 서버에 Supabase와 Next 앱을 올리고 Nginx로 묶어 직접 운영 환경을 구성했습니다.

### 셀프호스팅 로그 누적으로 인한 다운 대응

운영 중 사이트가 다운되어 확인해 보니 셀프호스팅 스택의 로그가 디스크를 채운 상태였습니다. 로그 출력과 보관 설정을 조정해 재발을 막았고, 매니지드 서비스를 쓰지 못한 선택의 운영 비용까지 직접 처리했습니다.

## 배포와 운영

2024년 8월 오픈 후 공개 URL로 운영됐습니다. 비정형 자료를 검색 가능한 데이터와 UI로 바꾸고, 배포 제약까지 직접 해결해 공개 포털로 완성한 사례입니다.
