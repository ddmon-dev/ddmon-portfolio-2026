## Overview

RGB Managers는 사내 프로젝트, 고객, 견적, 홈페이지 포트폴리오·문의를 하나의 관리 흐름으로 묶은 내부 백오피스입니다. 외주사가 게시판을 커스텀해 만든 PHP 레거시 도구를 대체하는 데서 출발해, 2023년부터 지금까지 직접 운영하며 계속 확장했습니다.

이 프로젝트의 가치는 사용자 수가 많은 데 있지 않습니다. 실제 사내 사용자가 있는 도구를 만들고, V1 운영 후 V2로 다시 설계하며, 이후 거래처 어드민 제작의 기반으로까지 이어진 점이 핵심입니다.

## 작업범위

- 사내 프로젝트·고객·견적·홈페이지 관리 흐름 재설계
- V1 Next.js 13 기반 신규 구축, V2 Next.js 15·shadcn 기반 재설계
- UI/디자인, FE, DB 설계, 마이그레이션, 배포 단독 수행
- MySQL 레거시 데이터 MongoDB 이관
- 견적서 작성·미리보기·PDF 출력 흐름 구현
- 2023년부터 현재까지 운영·유지보수

## 기획 의도

기존 사내 툴은 프로젝트 관리 게시판에 가까웠고, 홈페이지별 포트폴리오·문의 관리도 각 사이트 어드민에 흩어져 있었습니다. 처음에는 기존 필드 개편 요청이었지만, PHP 레거시를 계속 고치는 것보다 사내 운영 흐름을 다시 설계하는 편이 낫다고 판단했습니다.

V1을 운영한 뒤에는 단순 UI 개편이 아니라 향후 어드민 개발의 기반으로 다시 세웠습니다. shadcn 기반 V2에서 공통 레이아웃, 폼, 목록, 인증 구조를 정비했고, 이후 거래처 CMS·어드민 프로젝트에 이 구조가 이식됐습니다.

## Key Features

- 프로젝트 상태, 진행 단계, 담당자, 댓글 관리
- 고객 CRM, 엑셀 대량 업로드·다운로드
- 견적서 작성, 실시간 미리보기, PDF 출력
- 회사 홈페이지 포트폴리오·문의 통합 관리
- 사내 약 20명 사용, 견적 데이터 약 900건 누적

## 핵심 설계 결정

### 흩어진 홈페이지 어드민을 하나의 운영 백오피스로 통합

각 홈페이지가 따로 갖고 있던 포트폴리오·문의 관리를 RGB Managers 안으로 모아 관리 표면을 줄였습니다.

- RGB, Design RGB, RGB3D 등 여러 사이트의 운영 데이터를 한 백오피스에서 처리.
- 포트폴리오 등록, 문의 확인, 프로젝트 관리를 분리된 어드민 대신 같은 작업 공간으로 통합.
- 홈페이지 납품 구조를 내부 운영 도구와 연결해, 제작물 관리까지 사내 시스템 안으로 흡수.
- 사내 툴 하나를 이후 고객사 어드민 구조의 출발점으로 확장.

### 요구 변동성이 큰 데이터는 복제 격리하고, 안정된 UI는 공통화

백오피스는 모듈마다 요구사항이 자주 바뀌기 때문에, 모든 것을 하나의 제네릭으로 묶으면 오히려 변경 비용이 커진다고 봤습니다.

- 테이블, 폼, 사이드시트, 확인 모달, 페이지 레이아웃은 공통 컴포넌트로 재사용.
- 데이터 조회와 스키마는 모듈별로 복제해 커스텀 자유도 확보.
- 잘못된 추상화로 모듈별 예외가 컨테이너 안에 쌓이는 상황을 회피.
- 중복 관리 비용보다 변경 영향 범위를 모듈 안에 가두는 쪽을 선택.

### 고객 필드 정의를 단일 원천으로 두어 폼·검증·엑셀을 파생

고객 DB는 필드가 바뀔 때 폼, 검증, 업로드 양식, 다운로드 컬럼이 함께 바뀌어야 했습니다. 이를 하나의 필드 정의 배열에서 파생되도록 구성했습니다.

- 필드 정의 하나에서 입력 기본값, Zod 검증, 엑셀 헤더, 업로드 필수 여부, 다운로드 컬럼 파생.
- JavaScript 기반 프로젝트라 타입 시스템이 잡아주지 못하는 누락을 런타임 구조로 보완.
- 필드 변경 시 여러 파일을 동시에 고치다 빠뜨리는 shotgun surgery 위험 축소.
- 이후 TypeScript 기반 템플릿으로 옮겨가게 된 성장 지점.

### 비개발 사용자용 견적서를 단일 폼과 실시간 출력물로 연결

영업·운영진이 엑셀로 관리하던 견적서를 웹 안으로 옮기면서, 여러 단계 마법사보다 한 화면에서 작성과 출력 확인을 함께 하는 흐름을 택했습니다.

- 좌측 입력과 우측 실제 견적서 미리보기를 나란히 배치.
- 항목 그룹과 세부 항목을 2단 구조로 정리하고 드래그 정렬.
- 입력값이 곧 출력물에 반영되어, 작성자가 결과물을 보며 수정.
- 실사용 규모에서 리렌더 비용보다 작성 효율과 확인 가능성을 우선.

### 견적서 PDF는 디자인 충실도를 위해 서버 렌더로 처리

디자인 에이전시의 견적서는 문서 품질과 고해상도 출력이 중요했습니다. 클라이언트 캡처 방식은 해상도와 스타일 재현에 한계가 있어, 서버 헤드리스 브라우저 렌더를 선택했습니다.

- 화면 미리보기와 PDF가 같은 견적서 컴포넌트를 공유.
- 회사별 견적 포맷은 발행처 레지스트리로 분기.
- Puppeteer 서버 렌더의 무거움과 배포 의존성은 감수.
- 지금이라면 더 가벼운 react-pdf 기반 문서로 갈 수 있다는 성장 지점.

## 트러블슈팅 · 운영 한계 대응

### MySQL에서 MongoDB로 운영 중 데이터 이관

V1 구축 때 기존 MySQL 데이터를 MongoDB로 옮기며, 서비스를 완전히 멈추기보다 이관 중 신규 입력을 막는 방식으로 정합성을 확보했습니다. 수백~천여 건의 데이터를 변환해 적재했고, 새 구조에 맞지 않는 데이터는 별도 컬렉션에 읽기 전용으로 보존했습니다.

### 담당자 정보를 임베드에서 참조 구조로 정리

V1은 프로젝트 문서에 담당자 정보를 통째로 넣는 방식이라 회원 정보가 바뀌면 관련 프로젝트를 모두 고쳐야 했습니다. V2에서는 담당자 참조를 ObjectId로 정규화하고 조회 시 조인해 보여주는 방식으로 바꿔, 변경 전파 비용을 줄였습니다.

### 서버 PDF 렌더링의 한글 폰트와 브라우저 안정화

운영 서버에서 PDF 한글 폰트가 깨지는 문제를 만나 Pretendard 폰트를 직접 설치하고 캐시를 갱신했습니다. 저사양 서버에서 Chromium이 불안정한 문제도 실행 옵션과 프로세스 정리로 보완했습니다.

## 배포와 운영

2023년부터 사내 실사용 도구로 운영했고, 견적 데이터는 약 900건 누적됐습니다. 퇴사 후에도 유지보수 요청이 이어졌고, 이 프로젝트에서 정리한 V2 구조는 이후 고객사 어드민과 CMS의 기반으로 이식됐습니다.
