## Overview

알지비커뮤니케이션즈에서 전 직원 약 20명이 실무에 쓰는 사내 통합 백오피스입니다. 프로젝트 관리·고객 CRM·견적서, 그리고 회사 홈페이지마다 따로 있던 포트폴리오·문의 어드민을 한 시스템으로 모아, 2023년부터 지금까지 직접 운영해 왔습니다. 견적서가 약 900건 쌓이는 동안, 파일로 흩어져 있던 견적·고객 관리가 한곳으로 정리됐습니다.

외주사가 게시판을 커스텀해 만든 PHP 레거시 '플래너'를 다시 만든 것이 출발이었습니다. 첫 Next.js 프로젝트(V1)로 시작해 shadcn 기반 V2로 재설계했고, 이 V2가 이후 사내 어드민의 공통 기반이 되었습니다.

## 작업범위

- FE를 축으로, 백엔드 개발자 없이 화면 설계·DB·마이그레이션·배포까지 직접
- 기획 문서·시안 없이 레거시 도구와 사내 업무 흐름을 분석해 요구사항·화면 정의
- 7개 도메인 모듈 · V1(Next.js 13) → V2(Next.js 15·shadcn) 재설계
- MySQL → MongoDB 데이터 이관 + V2 스키마 정규화
- 2023년부터 3년 이상 단독 운영·유지보수

## 기획 의도

대표가 기존 플래너의 필드 개편을 요청한 것이 발단이었지만, PHP 레거시라 고쳐 쓰느니 다시 만드는 편이 낫다고 판단했습니다. 단순 이식이 아니라, 사내에 필요한 운영 도구를 계속 붙일 수 있는 단일 백오피스로 방향을 잡았습니다. 그때까지 회사 홈페이지 네 곳이 저마다 어드민을 따로 두고 있어 포트폴리오·문의 관리가 분산돼 있었는데, 그중 세 곳을 이 시스템 하나로 흡수하는 것이 핵심이었습니다.

V1을 약 1년 운영한 뒤 shadcn이 등장하자, 이 시스템을 향후 어드민 개발의 기반으로 다시 세우기로 했습니다. 그래서 V2는 단순 UI 교체가 아니라, 이후 프로젝트들이 그대로 가져다 쓸 컴포넌트·인증·구조를 의식하며 재설계했습니다.

## Key Features

- 프로젝트 관리: 상태·진행 단계, 담당자 배정, 프로젝트별 댓글
- 고객 CRM: 고객 이력 관리, 엑셀 대량 업로드/다운로드
- 견적서: 단일 폼 작성·실시간 미리보기, 회사별 포맷 PDF 출력
- 회사 홈페이지의 포트폴리오·문의 통합 관리

## 핵심 설계 결정

### V2를 이후 사내 어드민의 공통 기반으로 설계

V2를 일회성 도구가 아니라 이후 사내 어드민이 그대로 올라설 공통 기반으로 설계했고, 실제로 다른 프로젝트의 어드민으로 이어졌습니다.

- shadcn 위에 자체 컴포넌트·폼·레이아웃 구조를 갖춰, 새 어드민을 빠르게 얹을 수 있는 베이스로 구성.
- 이 시점에 어드민 인증을 Supabase Auth에서 NextAuth로 표준화했고, 이후 프로젝트들이 같은 인증 방식을 따랐습니다.
- 사내 도구 하나를 키운 경험이 다음 프로젝트의 출발점이 되는, 재사용 레버리지로 이어진 사례.

### 요구 변동성 기준 추상화 경계: 표현은 공통, 데이터는 모듈 복제

표현·상호작용 컴포넌트는 단일 소스로 추상화하되, 데이터 조회와 스키마는 모듈마다 복제해 커스텀하는 구조를 의식적으로 택했습니다. 백오피스는 요구사항 변동이 잦아, 하나의 제네릭이 모든 모듈을 흡수하면 모듈별 커스텀 자유가 떨어지고 비용이 폭발한다고 봤기 때문입니다.

- 테이블·폼·모달·사이드시트 등 공통 컴포넌트는 한 곳에서 정의해 화면 곳곳에서 재사용(폼 필드 18곳, 확인 모달 14곳, 페이지 레이아웃 13곳, 목록 12곳, 사이드시트 11곳).
- 자주 바뀌는 데이터·스키마는 레퍼런스 하나를 두고 모듈별로 복제해 커스텀했고, 변경의 영향 범위를 모듈 안에 가뒀습니다.
- 처음 검토한 '단일 제네릭으로 전부 커버' 안은 위 이유로 기각했습니다. 복제의 중복 비용을 잘못된 추상화의 경직성보다 싼 값으로 본 트레이드오프입니다.

### 단일 필드 정의(SSOT)로 폼·검증·엑셀·DB 일괄 파생

고객 DB의 필드 정의를 배열 하나에 모아, 입력 폼·검증·엑셀 업로드/다운로드·DB 저장이 모두 그 정의에서 파생되도록 설계했습니다. 당시 JavaScript로 시작한 선택이라 필드가 바뀔 때 여러 곳 중 한 곳만 빠져도 곧 버그여서, 단일 출처로 누락 자체를 차단하려는 결정이었습니다.

- 필드 정의의 메타 플래그로 채널별 노출 제어: 업로드 제외·필수·다운로드 제외를 한곳에서 선언.
- 같은 정의가 엑셀 양식의 헤더·검증 규칙과 화면 폼 기본값을 동시에 생성해, 양식과 화면이 어긋날 여지를 없앴습니다.
- 타입 시스템이 잡아줄 정합성을 런타임 구조로 대신 보장한 선택이자, 이후 TypeScript 전환의 동기가 된 한계 인식.

### 비개발 사용자용 견적서: 단일 폼 + 실시간 미리보기 통합

견적서 작성을 여러 단계로 쪼개지 않고, 한 화면의 단일 폼에 실제 출력될 견적서 미리보기를 나란히 붙였습니다. 그전까지 견적서를 엑셀 파일로 관리하던 비개발 사용자가 스크롤을 최소화하고 전체 내용을 한눈에 보며 빠르게 작성하도록 만드는 것이 목표였습니다.

- 작성 내용과 미리보기를 묶어, 입력하는 즉시 출력 모습으로 반영. 보이는 그대로 출력.
- 견적 항목은 그룹·항목 2단으로 드래그 정렬하고, 동적 배열의 키를 고정해 재정렬 시 꼬임을 막았습니다.
- 입력마다 미리보기를 다시 그리지만 실사용 규모에서 체감 지연이 없어, 선제 최적화보다 단순함을 유지했습니다.

### 디자인 충실도 위해 헤드리스 서버 렌더로 견적서 PDF 분리

견적서를 클라이언트에서 이미지로 캡처하지 않고, 화면을 그대로 직렬화해 서버의 헤드리스 브라우저로 렌더했습니다. 디자인 에이전시의 견적서라 디자인 충실도와 고해상도 출력이 중요한데, 처음 검토한 클라이언트 캡처 방식으로는 두 요건을 만족할 수 없었기 때문입니다.

- 화면 미리보기와 PDF가 같은 컴포넌트를 공유해, 보이는 견적서와 출력물이 구조적으로 일치.
- 회사별 견적 포맷은 발행처를 키로 컴포넌트를 골라내는 레지스트리로 분기해, 발행처가 늘어도 등록만 추가하면 됩니다.
- 서버 렌더라 무겁고 배포 환경에서 손이 많이 가는 비용을 감수한 선택으로, 지금이라면 더 가벼운 선언형 PDF로 갈 것입니다.

## 트러블슈팅 · 운영 한계 대응

### MySQL→MongoDB 무중단 이관과 담당자 참조 정규화

MySQL 레거시 데이터를 MongoDB로 옮길 때, 서비스를 멈추는 대신 이관 중 신규 입력을 막아 정합성을 확보했습니다. 수백~천여 건을 SQL 덤프에서 변환해 적재했고, 새 스키마에 맞지 않는 잔여 데이터는 별도 컬렉션에 읽기 전용으로 보존했습니다.

V2에서는 데이터 모델 자체를 손봤습니다. 프로젝트 목록 화면이 담당자 이름을 정확하고 일관되게 보여주려면, V1처럼 프로젝트 문서에 담당자 정보를 통째로 박아 두는 방식은 회원 정보가 바뀔 때마다 그 사람이 들어간 프로젝트를 전부 고쳐야 했습니다. 이를 참조 방식으로 바꿔 조회 시 조인으로 이름을 붙이도록 정리했고, 변환 스크립트가 진짜 원자성을 보장하지 않는 한계는 레거시 컬렉션을 남겨 원복 가능성을 확보하는 방식으로 감수했습니다.

### 서버 PDF 한글 폰트 부재 해결과 헤드리스 브라우저 안정화

사용자에게 견적서가 깨져 보이지 않게 하려다, 서버에서 PDF를 렌더할 때 운영 서버에 한글 폰트가 없어 글자가 깨지는 문제를 만났습니다. 원인을 폰트 부재로 진단해 서버에 폰트를 직접 설치하고 캐시를 갱신했으며, 폰트 로딩이 끝난 뒤 렌더하도록 했습니다. 저사양 서버에서 헤드리스 브라우저가 공유 메모리 부족·좀비 프로세스로 죽는 문제도, 실행 옵션을 조정하고 작업 후 프로세스를 확실히 정리하는 방식으로 안정화했습니다.

### 세션 만료 시 빈 세션 참조로 멈추던 클라이언트 경계 버그

세션이 만료된 뒤 사용자 정보가 빈 상태인데도 화면이 그 값을 직접 참조해 멈추는 버그가 있었습니다. 서버 검증을 통과한 뒤 클라이언트에서 빈 세션을 만지며 터지는 경계 문제로 진단해, 화면에서는 값 접근을 방어하고 클라이언트에서 세션 만료를 감지해 로그인 화면으로 자동 이동하도록 함께 고쳤습니다.

## 배포와 운영

사내 전용 백오피스라 로그인 벽 안에서 동작하며, 가비아 클라우드에 셀프호스팅해 운영합니다. 파일 스토리지는 검증된 기존 서버를 그대로 두고 앱만 옮기는 점진적 전환을 택했습니다.

2023년부터 모든 커밋을 직접 관리하며 큰 사고 없이 운영해 왔고, 견적·고객을 다루는 영업·운영진이 특히 관리하기 편해했습니다. 사용자 수는 많지 않지만 실사용자가 눈앞에 있어 피드백이 곧바로 반영되는, 그만큼 가장 오래 운영하며 키워 온 서비스입니다. 퇴사 후에도 유지보수를 맡아 지금까지 운영되고 있습니다.
