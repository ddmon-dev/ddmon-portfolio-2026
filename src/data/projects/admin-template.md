## 개요

이 프로젝트는 거래처 홈페이지의 어드민 화면을 풀스택으로 빠르게 구축하기 위해 제작하였습니다. 어드민 구축 시 반복되는 패턴을 구조화하여 신규 프로젝트 개발과 기존 프로젝트 유지보수 용이성을 중점으로 잡았습니다.

프로젝트 내 관리모듈을 추가하는 구조를 디자인 하는 것이 이 프로젝트의 핵심이었으며, 과도한 추상화가 모듈의 자율성을 침해하지 않도록 하기 위해 여러 패턴을 시도하여 적정선을 찾았습니다.

이를 기반으로 개인용 업무 관리 툴인 Work Manager를 비롯하여 솔라가드 건축용 필름 보증서 서비스 등의 고객사 프로젝트의 어드민을 개발하였습니다.

특히 이렇게 구조화된 설계와 문서화된 컨벤션, 가이드는 그 자체로 하네스 역할을 해주어 AI agent 활용시 개발의 정확도와 속도면에서 향상된 결과를 얻을 수 있었던 점이 흥미로웠습니다.

## 작업범위

- FSD 기반 `app`, `features`, `shared` 레이어 구조 정의
- `manage-modules` 기반 CRUD 모듈 구조와 공통 Server Actions 구현
- Next.js 라우트 그룹 기반 라우트 구조 설계
- 개발/유지보수 워크플로우 및 컨벤션 설계
- Supabase 마이그레이션 워크플로우 설계

## 핵심 설계 결정

### Manage modules 기반 관리 기능 개발 구조 설계

"디렉토리 복제에 의한 단위 관리 기능의 간편한 추가와 커스텀의 용이성"을 목표로 설계하였습니다.

어드민 프로젝트에서 각각의 관리 기능은 목록, 검색, 상세, 등록, 수정, 삭제 기능을 기본적으로 가져갑니다. `_base` 디렉토리에 이러한 흐름을 제공하는 인프라를 두고, 개별 모듈이 자신의 디렉토리에서 확장하는 패턴으로 설계했습니다. `_base`가 제공하는 기본적인 인터페이스 위에서 각 모듈들은 `CONFIG`와 `schema.ts`를 기준으로 공통 액션, 목록 조건, 폼 검증을 빠르게 연결합니다.

컴포넌트들은 과도한 추상화로 커스텀을 방해하지 않도록 분리하고, 모듈별로 파일을 복제해서 사용하는 것을 채택했습니다. 중복 코드와 파일은 늘어나겠지만 구조를 `_base`에 숨겨두는 패턴이 오히려 러닝커브를 늘리고 커스텀 및 유지보수를 어렵게 한다고 판단했습니다.

특히 서버액션에서 이 점이 두드러졌습니다. 초기에는 factory 패턴을 시도했으나, 실제 프로젝트에 적용하다 보니 함수 호출과 함수 인자의 추적이 혼잡해지는 문제가 발생했습니다. 따라서 공통 로직은 `_base`에 두고, 각 모듈의 액션 파일은 `CONFIG`를 넘기는 얇은 함수로 유지하는 방식으로 전환해 직관성을 챙겼습니다.

### Supabase 기반 프로젝트 초기화 및 유지보수 흐름 설계

Supabase 초기화 흐름은 PC에 프로그램을 설치하는 과정에 착안해 설계했습니다. 새 프로젝트를 시작할 때 앱을 구동하기 위한 필수 구조를 초기화하도록 마이그레이션을 구성했습니다. 그 외 기본 관리 테이블들은 별도의 마이그레이션 파일로 준비해두어 필요 시 적용할 수 있도록 하였습니다.

개발 중에는 로컬 Supabase CLI 기반으로 작업하도록 하고, 마이그레이션 기반으로 작업하도록 컨벤션을 설계했습니다. 이러한 구성으로 초기 세팅을 빠르게 가져가고, 개발 중에도 불필요한 의사결정을 줄여 기능 구현에 집중할 수 있는 구조를 의도했습니다.
