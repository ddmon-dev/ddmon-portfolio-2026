## Overview

거래처마다 새로 짜던 어드민 골격(목록·검색·페이지네이션·작성/수정)을 config 하나로 양산하는 재사용 기반으로 만든 본인 IP입니다. 파노라마 필름을 첫 Next.js 프로젝트로 끝낸 뒤, 반복되는 어드민을 매번 새로 짜는 대신 구조로 흡수하는 쪽이 더 큰 레버리지라고 봤습니다.

생고뱅 건축필름과 골프존(클라우드·스마트캐디)의 거래처 어드민, 그리고 본인 내부 도구까지 — 같은 엔진을 여러 곳에 이식하며 추상화가 실제로 재사용되는지 검증했습니다. 닫힌 납품작과 달리 공개 가능한 본인 코드라, 거기서 내린 설계 판단을 그대로 증명할 수 있는 자산이기도 합니다.

## 작업범위

- 단독 설계·구현 — FE 아키텍처부터 인증·DB 스키마까지
- 6개 도메인 모듈을 공통 CRUD 엔진 위에 config로 구성, 화면 표현은 공용 부품으로 조립
- 폼·검증 추상화와 권한 게이팅 — 화면이 요구하는 데이터를 끝까지 보장하려 DB 스키마·동시성 RPC까지
- 구조·설계 판단은 직접, 디테일 구현은 AI 페어
- AI가 내놓은 코드를 그대로 신뢰하지 않도록 단위 테스트 검증 레이어를 직접 설계
- 여러 거래처 어드민과 내부 도구에 이식, 실사용 피드백을 템플릿에 역으로 반영

## 기획 의도

파노라마 필름 이후 거래처마다 어드민의 목록·검색·작성/수정 골격이 똑같이 반복된다는 걸 확인하고, 새 관리 모듈을 며칠이 아니라 몇 시간 만에 세울 수 있는 재사용 기반을 만들기로 했습니다. 단순히 컴포넌트를 모아두는 게 아니라, 데이터 처리는 자동화하고 화면은 조립하는 구조를 목표로 했습니다.

그 과정에서 가장 많이 다룬 판단은 기능을 늘리는 일이 아니라 "어디까지 추상화하고 어디서 멈출지"였습니다. 아래 설계 결정 대부분이 그 경계를 어떻게 그었는지에 대한 기록입니다.

## Key Features

- 공지·뉴스·FAQ·팝업·문의·관리자 6개 도메인 모듈을 공통 CRUD 엔진으로 구성
- 선언형 폼 킷(React Hook Form Controller + zod) + 작성 중 이탈 방지 가드
- 검색·페이지·필터 상태를 URL에 보존하는 목록 — 새로고침·공유에도 유지
- 항목 순서변경, 소프트 삭제
- 관리자 인증·권한 게이팅(보호 라우트 + 액션 검증), 유휴 시 자동 로그아웃
- CKEditor 기반 본문 편집(이미지 업로드 포함), 문의 답변 메일 발송

## 핵심 설계 결정

### 데이터는 제네릭으로, 표현은 조립으로 — 추상화를 레이어별로 다르게 그었다

목록·검색·페이지네이션·작성/수정 같은 데이터 CRUD는 config 하나로 흡수하되, 폼·컬럼 같은 화면 표현은 일부러 제네릭화하지 않고 공용 부품 조립으로 남겼습니다.

- 모듈 액션은 `baseGetList(CONFIG, params)` 한 줄 — 테이블명을 `as const`로 고정하면 행 타입과 DTO가 자동 파생, 여섯 모듈의 CRUD가 하나의 제네릭 위 얇은 바인딩으로 수렴.
- config를 함수 인자가 아니라 모듈 스코프에 고정한 이유 = Server Action 경계는 zod 스키마·config 객체를 직렬화해 넘길 수 없음. 클라이언트에는 직렬화 가능한 파라미터만 노출.
- 표현 레이어(폼·컬럼·상세)는 모듈마다 본질적으로 다른 화면(모듈당 100~177줄)이라 제네릭화하면 메타데이터 유지보수가 더 비쌈 — "config만 더하면 화면"이 아니라 "데이터는 자동, 표현은 손으로"가 정직한 경계.
- 관리자 모듈은 조회는 공통 엔진을 그대로 쓰되 생성·수정·삭제만 따로 — 비밀번호 해시·중복키 분기·권한 보호처럼 공통 엔진이 모르는 규칙은 액션 단위로 떼어냄. 공통이 진짜 공통일 때만 추상화한다는 선.
- 처음엔 서버 로직을 팩토리 함수로 찍어냈지만 래퍼가 겹치며 이해 비용이 복제 비용보다 커져 걷어냄 — 복제 보일러플레이트는 늘었지만 러닝커브와 디버깅 비용이 내려간 의식적 트레이드오프.

### 작성 중 이탈 방지를 라이브러리 내부 동작까지 파고들어 설계했다

폼을 수정하다 시트를 닫거나 새로고침할 때 입력이 날아가지 않도록, React Hook Form의 내부 구독 방식까지 이용한 더티 가드를 만들었습니다.

- `formState.isDirty`는 effect가 아니라 렌더 단계에서 읽어야 RHF의 지연 프록시 구독이 켜진다는 점을 이용 — 그 값을 ref에 흘려 닫기 콜백이 항상 최신 dirty 상태를 참조하도록 연결.
- 자식 폼이 부모 시트에 가드 함수를 거꾸로 등록(제어 역전)해, 닫기 3경로(Esc·바깥 클릭·닫기 버튼)와 모드 전환, 브라우저 새로고침까지 한 곳에서 단일 가로채기로 수렴.
- 가드 동작은 단위 테스트 9개로 고정 — 등록·해제, 최신 상태 반영, 언마운트 시 리스너 정리.
- 트레이드오프: 렌더 중 상태를 읽는 건 라이브러리 내부 규약에 기댄 패턴이라, 유지보수자가 무심코 걷어내지 않도록 의도를 코드 주석으로 명시.

### 순서 기능 전체를 config 플래그 하나에 응집했다

모듈 config에 순서 기능 플래그 한 줄을 켜면 항목 순서 관련 동작 전체가 따라오도록, 한 플래그를 네 지점으로 퍼지게 설계했습니다.

- 정렬 기준 변경, 신규 항목 순번(최댓값+1, 삭제된 행 제외) 자동 부여, 목록에 순서 변경 UI 자동 주입까지 한 번에 연결.
- 검색 중에는 순서 UI를 자동으로 숨김 — 검색 결과의 불연속 순번에서는 인접 교환이 무의미해지는 탓.
- 두 행의 순번 교환은 개별 UPDATE 두 번이 아니라 단일 RPC가 CASE 한 문장으로 동시 처리 — 중간 상태가 없어 동시 편집에도 안전. UI는 이동 중 전역 잠금으로 보강.
- 트레이드오프: 한 줄로 켜지는 편의의 대가는 그 파급이 네 곳에 퍼진다는 것 — 연결 지점을 조회·생성·목록·교환으로 좁혀 응집하는 것으로 추적 비용을 관리.

### 같은 검증 규칙을 클라이언트와 서버가 한 벌로 쓰게 했다

입력 검증 스키마 하나를 클라이언트 폼과 서버 액션이 함께 구동하게 해, 검증이 두 곳에서 어긋나지 않도록 했습니다.

- 같은 zod 스키마가 클라이언트 `zodResolver`와 서버 `safeParse`를 동시에 구동 — 화면 전용 필드는 클라이언트 스키마에서만 확장하고, 서버 검증에는 행 컬럼만 넘김.
- 서버는 검증 통과분으로 값을 교체(parse-don't-validate) — 스키마에 없는 필드는 자동 탈락하므로, 관리자 생성 폼에 권한 필드를 정의하지 않은 것 자체가 권한 상승(mass-assignment) 차단.
- 비밀번호 변경은 클라이언트 검증으로 끝내지 않고 현재 비밀번호를 서버에서 다시 조회·대조, 수정 대상도 세션 사용자 본인 행으로 한정.
- 파노라마에서 비폼 UI를 명령형으로 폼에 이어 붙이던 방식을 걷어내, 이 템플릿에는 값을 명령형으로 밀어넣는 호출이 한 번도 없음 — 납품작(JS·명령형)에서 본인 자산(TypeScript·선언형)으로 넘어온 흔적.
- 트레이드오프: 부분 수정을 위해 스키마를 partial로 완화하는 경로는 순수 객체 스키마를 전제 — refine이 얽힌 스키마는 관리자 모듈처럼 액션 단위로 분리해 회피.

### 목록 상태를 화면이 아니라 URL에 두었다

검색어·페이지·필터를 React 상태가 아니라 URL 쿼리스트링에서 읽는 파생값으로 둬, 새로고침·공유·뒤로가기가 별도 코드 없이 동작하게 했습니다.

- 목록의 페이지·검색어·카테고리가 컴포넌트 상태가 아니라 URL에서 파생 — 상태의 단일 원천이 URL 그 자체.
- 잘못된 페이지 번호(주소 직접 조작 포함)는 클라이언트와 서버가 같은 공식으로 마지막 페이지로 보정하는 이중 클램핑.
- 테이블은 TanStack을 헤드리스로만 써(페이징·필터는 서버 책임), 라이브러리는 렌더 엔진으로 한정.
- 트레이드오프: URL을 단일 원천으로 삼으면 상태 변경마다 라우터 이동이 일어나므로, 검색과 페이지 초기화를 한 번의 이동으로 묶어 불필요한 네비게이션을 줄임.

### 권한을 레이아웃과 액션 양쪽에서 막았다

화면 진입은 레이아웃 게이트로, 실제 권한은 모든 서버 액션이 다시 확인하도록 — 권한 검증을 한 겹이 아니라 두 겹으로 깔았습니다.

- 보호 라우트 그룹 레이아웃이 진입을 막고(최고관리자 전용 영역은 별도 게이트), 서버 액션은 레이아웃을 거치지 않고 직접 호출될 수 있으므로 액션마다 인증을 다시 검증.
- NextAuth 세션 클레임은 모듈 타입 보강으로 컴파일 단계부터 타입 안전하게, 루트 레이아웃에서 세션을 미리 주입해 첫 렌더 깜빡임을 제거.
- 유휴 시간 자동 로그아웃과 세션 슬라이딩 갱신을 같은 상수로 묶어, 클라이언트 타이머와 서버 토큰 만료가 어긋나지 않게 동기화.
- 트레이드오프: 미들웨어로 중앙에서 강제하지 않아 신규 액션에서 검증 누락이 구조적으로 가능 — 동일 헬퍼로 단일화해 위험을 좁혔고, JWT 전략이라 권한 회수가 토큰 만료까지 지연되는 비용도 인지한 선택.

## 트러블슈팅 · 운영 한계 대응

### 시트 안의 에디터가 포커스를 빼앗기는 충돌

Radix 시트의 포커스 트랩과 CKEditor 자체 포커스 관리가 충돌해, 시트 안에서 에디터에 포커스가 잡히지 않았습니다. 시트를 비모달로 열고 스크롤 잠금을 수동으로 처리하며, 바깥 클릭으로 닫히는 동작만 별도 오버레이로 살려 — 포커스 트랩은 끄되 닫기 UX는 유지하는 방식으로 풀었습니다. 라이브러리 두 개의 내부 동작(포커스 트랩·스크롤 잠금)을 이해해야 나오는 우회입니다.

### 취소할 수 없는 외부 발송을 먼저 확정하고 DB를 사후 보정하도록 순서를 고정했다

문의 답변은 관리자에게 "메일 발송됨"과 "답변 저장됨"이 한 동작처럼 보여야 하는데, 외부 SMTP 발송은 DB 트랜잭션으로 감쌀 수 없습니다. 그래서 메일을 먼저 보내고 성공한 뒤에만 답변을 저장·상태 변경하도록 순서를 고정했습니다 — 메일은 취소할 수 없는 외부 부수효과라 먼저 확정하고, DB는 사후 보정이 가능하다는 판단입니다. 메일은 갔는데 저장이 실패하면 불일치가 남을 수 있다는 비원자성은 인지한 선택이고, "메일 실패 시 저장하지 않는다"는 순서 불변식은 단위 테스트로 고정했습니다.

## 이식과 재사용

이 템플릿은 그 자체로 배포되는 서비스가 아니라, 실제 거래처 어드민이 올라서는 기반입니다. 생고뱅 건축필름 어드민을 시작으로 골프존 클라우드·스마트캐디 어드민, 그리고 견적·프로젝트·일정을 다루던 초기 내부 업무 도구까지 — 단순 게시판을 넘어 복합 도메인을 다루는 화면들이 같은 엔진 위에 올랐습니다.

핵심은 책상 위 추상화가 아니라 실제 이식으로 검증된 기반이라는 점입니다. 첫 거래처에 적용한 직후 실사용에서 나온 버그·개선점을 템플릿에 되돌려 반영했고, 이후 거래처를 거치며 폼 가드·검증·문의·팝업 같은 기능이 더해졌습니다. 그 덕에 간단한 어드민은 하루이틀, 핵심 기능이 많은 어드민도 한 주 안팎으로 새로 세울 수 있었습니다.

닫힌 납품작은 소스를 보여드릴 수 없지만, 이 템플릿은 직접 만든 공개 가능한 자산이라 거래처 어드민에서 내린 것과 같은 설계 판단을 코드로 그대로 확인할 수 있습니다.
