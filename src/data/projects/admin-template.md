## Overview

Next.js 어드민 템플릿은 거래처마다 반복되던 목록·검색·페이지네이션·작성/수정 어드민을 빠르게 세우기 위해 만든 본인 IP입니다. 닫힌 납품작의 소스를 공개할 수 없는 대신, 같은 설계 판단을 코드로 보여줄 수 있는 포트폴리오의 코드 영수증입니다.

이 프로젝트의 핵심은 “config 하나로 모든 화면을 만든다”가 아닙니다. 데이터 CRUD는 제네릭으로 줄이고, 폼·컬럼·상세처럼 도메인별 차이가 큰 표현 레이어는 조립으로 남기는 경계를 정한 점입니다.

## 작업범위

- Next.js 16, TypeScript, Supabase 기반 어드민 템플릿 설계·구현
- 공통 CRUD 액션, 목록, 폼, 검증, 권한 게이팅 구조 작성
- React Hook Form + Zod 기반 선언형 폼과 서버 검증 연결
- NextAuth 기반 관리자 인증과 보호 라우트 구성
- 순서 변경, 파일 업로드, CKEditor, 문의 답변 등 운영 기능 구현
- 단위 테스트와 거래처 어드민 이식

## 기획 의도

파노라마 필름 이후 거래처마다 어드민의 목록, 검색, 작성/수정 골격이 반복된다는 것을 확인했습니다. 매번 새로 짜는 대신, 데이터 처리는 자동화하고 화면 표현은 공용 부품으로 조립하는 재사용 기반을 만들기로 했습니다.

처음에는 더 많은 것을 팩토리화하려 했지만, 래퍼가 겹칠수록 이해 비용이 올라갔습니다. 그래서 “데이터 CRUD는 제네릭, 표현은 모듈별 조립”이라는 경계로 되돌렸습니다. 코드 작성 비용은 AI가 많이 줄여주지만, 읽고 판단하는 비용은 여전히 사람이 부담하기 때문입니다.

## Key Features

- 6개 도메인 모듈의 목록·검색·작성/수정 CRUD
- React Hook Form Controller + Zod 기반 선언형 폼
- 작성 중 이탈 방지 더티 가드
- URL 기반 검색·페이지·필터 상태
- 관리자 인증, 보호 라우트, 액션 재검증
- 순서 변경, 소프트 삭제, CKEditor, 문의 답변 메일

## 핵심 설계 결정

### 데이터 CRUD는 제네릭, 화면 표현은 조립으로 남김

목록 조회와 생성·수정·삭제 같은 데이터 작업은 공통화하되, 폼·컬럼·상세 화면은 모듈별로 직접 조립하게 했습니다.

- 모듈 액션은 `baseGetList(CONFIG, params)` 같은 얇은 바인딩으로 수렴.
- config 객체와 zod 스키마는 Server Action 직렬화 경계를 넘기기 어려워 모듈 스코프에 고정.
- 테이블명을 `as const`로 고정해 행 타입과 DTO 타입이 자동 파생.
- 표현 레이어는 모듈마다 100줄 이상 차이가 나므로 과한 메타데이터화보다 수기 조립이 더 읽기 쉬움.

### RHF 지연 프록시를 이해하고 작성 중 이탈 방지 가드를 설계

폼을 수정하다가 시트나 브라우저를 닫을 때 입력이 사라지지 않도록, React Hook Form의 구독 방식까지 고려해 더티 가드를 만들었습니다.

- `formState.isDirty`는 렌더 단계에서 읽어 RHF lazy proxy 구독 활성화.
- 최신 dirty 상태는 ref에 미러링해 닫기 콜백의 stale closure 방지.
- 자식 폼이 부모 시트에 가드 함수를 등록하는 제어 역전 구조.
- Esc, 바깥 클릭, 닫기 버튼, 모드 전환, 새로고침을 하나의 가드 경로로 수렴.

### config 플래그 하나가 정렬·생성·UI·동시성까지 연결

순서 변경 기능은 단순 버튼 추가가 아니라 목록 정렬, 신규 순번, 검색 상태, DB 교환 로직이 함께 움직여야 했습니다.

- `enableReorder` 한 줄로 `sort_order` 정렬, 신규 항목 max+1, 순서 컬럼 자동 주입.
- 검색 중에는 순서 UI를 숨겨 불연속 결과에서 잘못된 인접 교환을 막음.
- 두 행의 순번 교환은 개별 UPDATE 두 번이 아니라 단일 RPC의 CASE 문으로 처리.
- UI 전역 잠금과 DB 원자 교환을 함께 둬 동시 편집 위험 축소.

### 클라이언트·서버 검증을 같은 zod 스키마로 일원화

입력 검증이 화면과 서버에서 따로 움직이면 시간이 갈수록 어긋납니다. 같은 스키마를 양쪽에서 쓰되, 화면 전용 필드는 분리했습니다.

- 같은 zod 객체가 클라이언트 `zodResolver`와 서버 `safeParse`를 구동.
- 서버는 검증 통과 결과인 `result.data`만 저장해 스키마 밖 필드를 자동 탈락.
- 관리자 생성 스키마에 `super_admin`을 넣지 않아 mass-assignment를 구조적으로 차단.
- 파노라마의 명령형 폼 연결에서 벗어나 Controller 기반 선언형 입력으로 재설계.

### 목록 상태를 URL에 두어 새로고침·공유·뒤로가기를 기본 동작으로 만듦

검색어, 페이지, 필터를 React 상태가 아니라 URL 쿼리에서 파생되게 했습니다.

- 목록 상태의 단일 원천을 URL로 고정.
- 검색 시 페이지 초기화까지 같은 라우터 이동으로 처리.
- 잘못된 페이지 번호는 클라이언트와 서버가 같은 공식으로 마지막 페이지 보정.
- TanStack Table은 헤드리스 렌더 엔진으로만 사용하고 페이징·검색 책임은 서버에 둠.

### 레이아웃 게이트와 액션 재검증으로 권한 경계를 이중화

화면 진입은 보호 라우트가 막고, 실제 데이터 변경은 각 Server Action이 다시 검증하도록 했습니다.

- 보호 레이아웃이 직접 URL 접근을 차단.
- Server Action은 레이아웃을 거치지 않고 호출될 수 있으므로 액션마다 인증 재검증.
- NextAuth 세션 클레임을 타입 보강해 관리자 권한을 타입 안전하게 사용.
- 유휴 로그아웃과 세션 갱신을 같은 상수로 묶어 UX와 인증 만료 흐름을 맞춤.

## 트러블슈팅 · 운영 한계 대응

### Radix 시트와 CKEditor 포커스 관리 충돌

Radix Sheet의 포커스 트랩과 CKEditor의 자체 포커스 관리가 충돌해 시트 안에서 에디터에 포커스가 잡히지 않았습니다. 시트를 비모달로 열고 스크롤 잠금은 수동 처리했으며, 바깥 클릭 닫힘만 별도 오버레이로 살려 에디터 입력과 닫기 UX를 함께 유지했습니다.

### SMTP 발송과 DB 저장의 비원자성 수용

문의 답변은 메일 발송과 DB 상태 변경이 한 동작처럼 보여야 하지만, 외부 SMTP 발송은 DB 트랜잭션으로 감쌀 수 없습니다. 취소할 수 없는 외부 부수효과인 메일을 먼저 확정하고, 성공한 뒤에만 답변 저장과 상태 변경을 수행했습니다.

## 이식과 재사용

이 템플릿은 생고뱅 건축필름, 골프존 클라우드, 골프존 스마트캐디 어드민과 초기 내부 도구에 이식됐습니다. 중요한 점은 같은 골격이 실제 거래처 도메인에 여러 번 적용되며, 첫 실사용 피드백이 다시 템플릿으로 되돌아왔다는 점입니다.

거래처 소스는 공개할 수 없지만, 이 템플릿은 공개 가능한 본인 코드입니다. 납품 프로젝트에서 반복해온 어드민 설계 판단을 코드로 확인할 수 있는 근거로 둡니다.
