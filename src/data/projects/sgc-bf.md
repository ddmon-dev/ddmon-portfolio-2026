## Overview

솔라가드가 건넨 엑셀 한 장 분량의 기획안에서 요구사항을 직접 정의해, 고객이 보증서를 조회·발급받는 공개 사이트와 이를 운영하는 어드민 — 두 개의 Next.js 앱을 2주 만에 단독으로 설계·배포한 건축필름 품질보증서 시스템입니다. 시공된 필름에 부착된 QR 스티커의 보증번호로 고객이 본인 보증서를 찾아 PDF로 발급·인쇄하고, 관리자는 발급·이력·제품·시공점을 관리합니다.

공개 영역과 관리 영역을 앱 단위로 분리하고, 거래처 어드민에 여러 번 이식해 온 본인 어드민 템플릿(TypeScript·Next.js 16) 위에 보증서 도메인을 올렸습니다. 초기 납품작에서 출발해 타입스크립트·선언형으로 정제해 온 어드민 기반을, 공개와 관리를 가른 2앱 구조로 다시 한번 끌어올린 작업입니다. 현재 sgcbf.co.kr에서 운영 중입니다.

## 작업범위

- 2주 · 1인 단독 — 공개 조회 사이트 + 어드민, 2개 Next.js 앱
- 솔라가드가 제공한 엑셀 한 장 분량 기획안에서 요구사항·화면을 정의
- 보증서 PDF 디자인 시안만 제공받고, 두 앱의 화면·UX는 설계
- 본인 어드민 템플릿 위에 보증서 도메인 구축 — Supabase 스키마·공개 API까지
- 설계·정책·검증 판단은 직접 결정, 구현은 AI 페어로 가속
- 배포와 운영

## 기획 의도

솔라가드 건축필름의 보증서 업무를 웹으로 옮기는 프로젝트였고, 받은 자료는 엑셀 한 장 분량의 기획안과 보증서 디자인 시안이 전부였습니다. 그래서 조회·발급의 흐름과 관리자 업무를 요구사항으로 정의한 뒤, 고객과 상담하며 정책을 확정했습니다. 특히 "보증번호를 무엇으로 삼을지"는 시공 현장의 QR 스티커 운영 방식을 함께 따져 잡은 정책입니다.

구현을 밑바닥부터 짜기보다, 거래처 어드민에 여러 번 이식해 온 본인 어드민 템플릿을 기반으로 보증서 도메인을 빠르게 올리는 쪽을 택했습니다. 짧은 일정 안에서도 공개 사이트와 어드민을 완성해 배포할 수 있던 이유입니다.

## Key Features

- 보증번호 또는 이름·연락처로 보증서 조회 (공개)
- 본인확인 후 보증서 PDF 발급·인쇄·다운로드
- 어드민 보증서 발급 — 제품·시공점 선택 시 연관 정보 자동 입력
- 엑셀 업로드 기반 대량 발급
- 제품(필름·VLT 옵션·보증기간)·시공점·발급 이력 관리
- 발급자별 데이터 격리 — 관리자는 본인 발급분만 조회

## 핵심 설계 결정

### 공개와 관리를 두 앱으로 갈라, 공개 앱에서 DB 접근을 걷어냈다

불특정 다수가 드나드는 공개 조회 사이트와 관리자 어드민을 한 앱에 합치지 않고, 별도의 Next.js 앱으로 분리했습니다. 공개 화면이 데이터베이스 자격증명이나 민감한 환경변수를 들고 있지 않게 하는 것을 1순위 원칙으로 삼았습니다.

- 공개 사이트는 자체 DB 없이 어드민이 노출한 공개 API만 호출 — 데이터의 단일 원천은 어드민 한 곳.
- 본인확인은 공개 앱이 아니라 어드민 서버가 권위적으로 수행 — 보증번호와 연락처가 맞고 비활성·삭제되지 않은 보증서만 발급.
- 공개 앱은 발급 권한이 없는 얇은 클라이언트 — 화면이 약속한 결과(본인 보증서)를 끝까지 책임지는 권위는 어드민 서버에 둠.
- 대가는 두 앱의 계약을 타입 공유 없이 수기로 맞추고 CORS 경계를 직접 관리해야 한다는 것 — 공개 영역의 공격면을 줄이는 값으로 수용.

### 같은 보증서 문서를 서버와 클라이언트 양쪽 렌더 경로에서 한 벌로 재사용했다

어드민에서 미리 보고 인쇄·다운로드하는 경로와, 공개 사이트에서 본인확인 후 발급받는 경로가 서로 다른 문서로 갈라지지 않도록 — 하나의 React 보증서 컴포넌트를 양쪽이 함께 쓰도록 설계했습니다. 발급 산출물이 어디서 나오든 한 글자도 달라지면 안 된다는 게 증서라는 도메인의 요구였습니다.

- 어드민은 클라이언트에서 문서를 렌더해 미리보기·인쇄·다운로드, 공개 발급은 어드민 서버가 같은 컴포넌트를 렌더해 PDF로 스트리밍 — 출력 경로는 여럿, 문서 정의는 하나.
- 개인정보 마스킹을 데이터가 아니라 출력(PDF 렌더) 단계에 둠 — 어느 경로로 발급되든 고객 이름·주소가 같은 규칙으로 가려짐.
- 보증 기한처럼 화면에 저장돼 있지 않은 값은 시공일과 보증 연수에서 문서가 직접 파생 — 발급 시점 사실로 고정.
- 대가는 한 문서를 두 경로가 공유하는 만큼, 그 컴포넌트를 서버·클라이언트 어느 쪽에서도 렌더되도록 환경 비의존으로 유지해야 한다는 것.

### 발급된 보증서는 마스터가 바뀌어도 변하지 않도록, 발급 시점 값을 박제했다

보증서가 참조하는 제품·시공점 정보를 외래키로 묶지 않고, 발급 시점의 값을 보증서 레코드에 그대로 복사해 저장했습니다. 보증서는 발급 당시의 사실을 고정해야 하는 증서라, 데이터 정규화보다 시점 불변성이 우선이라고 봤습니다.

- 제품명·VLT·보증기간·시공 정보를 발급 시점 스냅샷으로 저장 — 이후 제품·시공점 마스터가 수정·단종돼도 이미 발급된 보증서는 그대로.
- 발급된 보증서는 마스터를 다시 참조하지 않음 — 마스터 변경이 재조회·재발급에서 과거 사실을 덮어쓰지 않도록.
- 외래키 정규화를 포기한 대가는 마스터 변경이 기발급분에 반영되지 않는다는 것 — 증서의 성격상 오히려 요구에 부합.
- 발급 폼에서 마스터를 고르면 연관 값이 자동 입력 — 값을 복사 저장하면서도 손입력 부담은 없음.

### 보증번호의 권위를 시스템이 아니라 현장의 QR 스티커에 두었다

보증번호를 시스템이 자동 채번하지 않고, 시공 현장에서 필름에 붙는 QR 스티커의 난수를 그대로 보증번호로 입력받도록 정했습니다. 보증서의 식별자는 화면이 아니라 고객이 손에 쥔 물리 산출물과 일치해야 한다는 도메인 판단이었고, QR 운영 방식을 고객과 상담해 확정한 정책입니다.

- 자동 채번(연도 + 일련번호 등) 대신 현장 스티커를 권위로 — 고객이 스티커 번호로 바로 조회하도록.
- 시스템의 책임은 채번이 아니라 유일성 보장 — 발급 시 중복 번호 차단.
- 대량 발급에서는 파일 안의 중복과 이미 발급된 번호의 중복을 각각 검사 — 수기 입력으로 같은 번호가 두 번 들어가는 것을 막음.
- 대가는 번호 생성을 사람 손에 맡긴다는 것 — 그래서 검증을 입력 단계로 끌어와 오류를 조기에 차단.

### 운영자가 틀린 데이터를 올릴 수 없도록 대량 발급을 검증 플로우로 설계했다

보증서를 엑셀로 한 번에 올리는 대량 발급에서, 업로드 기능 자체보다 "잘못된 데이터가 발급되지 않게 막는" 검증 흐름을 먼저 설계했습니다. 운영자가 수십 건을 한 번에 넣을 때 깨지는 지점은 업로드가 아니라 데이터 정합이라고 봤기 때문입니다.

- 업로드한 각 행을 등록된 제품·시공점과 대조 — 없는 시공점이나 그 제품에 없는 VLT 옵션은 발급 차단.
- 검증 결과를 표 위에서 바로 고치는 인라인 프리뷰 — 셀을 수정할 때마다 재검증하고 연관 값(보증기간 등)을 다시 채움.
- 클라이언트 검증을 통과해도 서버에서 한 번 더 검증 — 화면을 우회한 입력 차단.
- 부분 성공을 허용하지 않고 일괄 삽입 한 번으로 처리 — 한 건이라도 틀리면 운영자가 파일을 고쳐 다시 올리도록. 절반만 발급돼 무엇이 누락됐는지 쫓는 것보다, 증서 발급에선 전부-아니면-전무가 안전하다고 판단.

## 배포와 운영

공개 사이트와 어드민을 단독으로 완성해 배포했고, 현재 sgcbf.co.kr에서 운영 중입니다. 엑셀 한 장짜리 자료에서 요구사항 정의부터 배포까지 전 범위를 책임진 작업으로, 거래처마다 이식해 온 본인 어드민 템플릿이 새 도메인을 짧은 일정 안에 올리는 기반으로 실제 작동하는지 다시 검증한 사례이기도 합니다.
