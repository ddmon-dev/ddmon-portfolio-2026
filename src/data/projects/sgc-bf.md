## Overview

솔라가드 건축필름 보증서 시스템은 현장 QR 스티커의 보증번호로 고객이 보증서를 조회·발급하고, 관리자가 제품·시공점·발급 이력을 운영하는 웹 서비스입니다. 공개 조회 사이트와 어드민을 분리해 2주 안에 단독으로 설계·배포했습니다.

이 프로젝트의 핵심은 새 기능을 많이 만든 것이 아니라, 보증서라는 문서가 가져야 하는 불변성과 공개 조회 흐름을 짧은 일정 안에서 안전한 구조로 정리한 점입니다. 기존 어드민 템플릿을 기반으로 반복 골격을 줄이고, 고유한 문제인 보증서 발급·PDF·대량 업로드 검증에 집중했습니다.

## 작업범위

- 공개 보증서 조회 사이트와 운영 어드민 2개 앱 개발
- 엑셀 한 장 분량 기획안에서 조회·발급·관리 흐름 정의
- 보증서 PDF 시안 기반 react-pdf 문서 구현
- 제품·시공점·보증서·발급 이력 데이터 구조 설계
- 공개 API, 본인확인 게이트, 대량 발급 검증 흐름 구현
- 배포 및 운영 인수

## 기획 의도

고객에게 제공된 QR 스티커의 보증번호를 기준으로, 본인 보증서를 조회하고 PDF로 발급받는 흐름이 필요했습니다. 자동 채번 시스템을 만드는 대신 현장에서 이미 쓰이는 물리 스티커를 식별자의 권위로 삼고, 시스템은 그 번호의 유일성과 발급 상태를 관리하는 역할로 좁혔습니다.

짧은 일정 안에서 밑바닥부터 모든 어드민을 새로 짜는 것은 비효율적이었습니다. 그래서 반복되는 관리 화면은 기존 어드민 템플릿을 활용하고, 이 프로젝트에서만 필요한 보증서 도메인, 공개 조회 흐름, PDF 발급, 엑셀 대량 검증에 설계 시간을 집중했습니다.

## Key Features

- 보증번호 또는 이름·연락처 기반 보증서 조회
- 본인확인 후 보증서 PDF 미리보기·인쇄·다운로드
- 제품·VLT 옵션·시공점·보증기간 관리
- 엑셀 기반 보증서 대량 발급과 인라인 오류 수정
- 관리자별 발급 데이터 격리

## 핵심 설계 결정

### 공개 조회 앱을 DB 없는 얇은 클라이언트로 분리

불특정 사용자가 접근하는 공개 사이트와 운영 어드민을 한 앱에 넣지 않고, 공개 앱은 어드민 API만 호출하는 얇은 클라이언트로 분리했습니다.

- 공개 앱은 DB 자격증명이나 운영용 환경변수를 갖지 않음.
- 데이터의 단일 원천은 어드민 앱과 그 DB로 고정.
- PDF 발급 전 본인확인은 어드민 서버가 번호와 연락처를 다시 검증.
- 대가는 두 앱 사이 API 계약과 CORS 경계를 직접 관리해야 한다는 점.

### 같은 보증서 컴포넌트를 서버·클라이언트 발급 경로가 공유

어드민 미리보기, 공개 사이트 PDF 발급, 다운로드, 인쇄 경로가 서로 다른 문서가 되지 않도록 보증서 문서 정의를 하나로 유지했습니다.

- 단일 react-pdf 보증서 컴포넌트를 클라이언트 미리보기와 서버 PDF 스트리밍에 함께 사용.
- 고객 이름·주소 마스킹, 보증기한 계산, 문서 레이아웃 규칙을 한 문서 정의에 집중.
- 보증서 시안은 고객사가 제공했지만, 이를 PDF primitive와 서버/클라이언트 렌더 양쪽에서 재현.
- 환경 양쪽에서 렌더 가능한 컴포넌트로 유지해야 하는 비용을 수용.

### 보증서는 마스터 참조보다 발급 시점 스냅샷을 우선

보증서는 나중에 제품명이나 시공점 정보가 바뀌어도 발급 당시의 사실을 유지해야 하므로, 마스터를 계속 참조하지 않고 발급 시점 값을 보증서 레코드에 복사했습니다.

- 제품명, VLT, 보증기간, 시공점 정보를 발급 시점 스냅샷으로 저장.
- 마스터 수정·단종이 이미 발급된 보증서 내용을 바꾸지 않음.
- 외래키 정규화를 일부 포기하는 대신 증서의 시점 불변성을 확보.
- 발급 폼에서는 마스터 선택 시 관련 값 자동 입력으로 운영자 입력 부담 완화.

### 보증번호 생성보다 현장 QR 운영과의 일치를 우선

보증번호를 시스템이 새로 만들지 않고, 시공 현장에서 붙이는 QR 스티커 난수를 그대로 보증번호로 입력받도록 정했습니다.

- 고객이 실제로 손에 쥔 QR 스티커 번호와 시스템 조회 번호가 일치.
- 시스템의 책임은 채번이 아니라 유일성 검증과 발급 상태 관리.
- 수기 입력과 엑셀 대량 발급 모두에서 중복 번호를 사전에 차단.
- 번호 생성이 사람 손에 남는 비용은 입력 단계 검증 강화로 보완.

### 엑셀 대량 발급은 업로드보다 검증 흐름을 먼저 설계

운영자가 여러 보증서를 한 번에 발급할 때 문제는 업로드 자체보다 잘못된 데이터가 조용히 발급되는 것입니다.

- 엑셀 행을 등록된 제품·VLT 옵션·시공점과 교차 검증.
- 검증 실패 행은 표 위에서 바로 수정하고 수정 즉시 재검증.
- 파일 내부 중복과 기존 발급 번호 중복을 각각 검사.
- 한 건이라도 틀리면 부분 성공 없이 다시 확인하게 해 증서 누락 추적 비용을 줄임.

## 배포와 운영

현재 sgcbf.co.kr에서 운영 중입니다. 추가 운영 수치가 큰 프로젝트는 아니지만, 본인 템플릿을 실제 거래처 도메인에 빠르게 이식하고, 보증서 발급이라는 고유한 업무 흐름을 2앱 구조와 PDF 발급 경계로 정리한 사례입니다.
